# V1.x Real Runtime Implementation Plan

This page is the English entry for the V1.x real runtime implementation plan. The full-detail source remains the Simplified Chinese document: [V1.x真实运行时能力补齐实施计划.md](../../zh-CN/planning/V1.x真实运行时能力补齐实施计划.md).

The plan is based on the [V1.x Incomplete Feature Inventory](v1.x-incomplete-feature-inventory.md). It turns the remaining V1.x gaps that still sit at contract, diagnostic, plan-first, in-memory, simulated, or controlled execution boundaries into a sequence of small implementation releases.

Current progress: V1.6 is complete through V1.6.5-alpha. It has landed the
durable backend manifest, layout, read-only verification, migration-lock
baseline, filesystem EventLog, durable EventBus dead-letter redrive, durable
task snapshots through `--durable-backend <path>`, `FileSystemAuditSink`
persistence under `audit/` with trace-id lookup, filesystem artifact metadata
hardening with content type, retention, legacy compatibility, stable
corrupt-metadata errors, runtime recovery scanning, event redrive checkpoints,
durable `runtime.recovered` audit records, clean-start/restart/corrupt-store
smoke, `inspect durable` backend diagnostics, and the
`REL-DURABLE-RECOVERY-001` and `REL-DURABLE-DIAGNOSTICS-001` release gates.
V1.7.4-alpha has now advanced the real Lua VM line by adding a restricted
`mlua`/Lua 5.4 VM adapter, real `on_event(ctx)` execution, stable syntax/runtime
error mapping, compatibility fallback for legacy static scripts, read-only Lua
context tables, `ctx.host.log/audit`, `ctx.tools.call`, Lua wall-clock timeout,
instruction budget, cancellation token, memory budget, shadow-load health
checks, scheduler generation route gating, generation drain evidence, rollback
audit evidence, and the
`REL-LUA-VM-EXECUTION-001`, `REL-LUA-HOST-BINDINGS-001`, plus
`REL-LUA-RESOURCE-LIMITS-001` and `REL-LUA-HOT-RELOAD-001` release gates.
Scheduler backoff dispatch, daemon-driven hot-reload orchestration, provider
process recovery, and complete runtime state restoration remain planned
follow-up slices.
V1.8 development has started after that alpha checkpoint: V1.8.1 wires
manifest-derived stdio/http adapter runners into `AdapterRuntime`, and V1.8.2
adds controlled MCP JSON-RPC stdio tool calls with allowlist, timeout,
output-limit, and protocol-error gates. V1.8.3 adds an MCP session registry,
health/shutdown/orphan cleanup, stream-abort boundary, explicit-tool server
gate, and stable MCP audit actions. V1.8.4 adds a controlled Skill workflow
runner with manifest input-schema gates, isolated working directories,
manifest allowlisted process execution, a managed `codex_skill` runner for the
sample adapter, filesystem artifact evidence, timeout/failure coverage, and
credential redaction. Real OS process supervision, richer Skill output-schema
validation, and production policy domains remain planned follow-up slices.
V1.13.3 has since landed the first provider admission baseline: adapter
manifests now feed `max_concurrency`, rate-limit, and circuit-breaker settings
into the in-memory provider supervisor; retryable admission failures expose
backoff hints, circuit-open health is recorded in the provider process table,
and half-open probes are allowed after the configured recovery window. V1.13.4
adds bounded provider stream summaries, redacted artifact sinks, and chunked
HTTP response reads. V1.13.5 adds a filesystem provider process table,
daemon-start provider execution-state recovery, interrupted stale active
sessions, and explicit retryable restart backoff evidence. V1.13.6 adds the
bounded MCP `http://` JSON-RPC client/auth boundary with manifest
`mcp.endpoint`/`mcp.headers`, provider session auth headers, allowlist-before-RPC,
and missing-auth policy errors. V1.13.7 adds a repo-local MCP compatibility
matrix fixture and required `REL-MCP-COMPAT-001` release gate for stdio/HTTP
transport, tool schema, stream lifecycle abort/cleanup, dangling-session, and
explicit server-surface evidence. V1.13.8 adds required
`REL-PROVIDER-SUPERVISION-001` release gate for the current controlled provider
supervision baseline while keeping OS process supervision explicitly out of
scope. Real OS process supervision, MCP production streaming/TLS/external-server
compatibility, and OS credential vault integration remain planned follow-up
slices.
V1.14.1 starts the destructive-restore wave by landing a plan-only staged
mutation planner. Restore apply plans can now carry copy/delete/replace steps,
target-root evidence, affected paths, deterministic preflight hashes, previews,
and rollback manifests. The CLI exposes this as optional `mutation_plan` JSON
while preserving the existing `mutation_executed:false` contract; real file
mutation and rollback apply are split across the later V1.14.2/V1.14.3 slices.
V1.14.2 then adds the gated staged file mutation engine: after confirmation,
artifact evidence, policy approval, apply lock, health, and staged steps all
pass, restore apply can execute copy/delete/replace operations in the target
tree, write a transaction log for each step, report `status:"applied"` with
`mutation_executed:true`, or stop with `rollback_required`. V1.14.3 then adds
`restore rollback`: it accepts only a rollback-required transaction log with a
matching staged preflight hash, restores committed copy/delete/replace steps in
reverse order from pre-restore archive entries, writes a rollback transaction
log, and reports `rollback_executed:true` only after reverse mutation commits.
V1.14.4 then adds additive operator confirmation output for `restore apply
--dry-run`, `restore apply`, and `restore rollback`, exposing confirm token,
plan id, target root, affected count, `apply_allowed`, `mutation_planned`,
`mutation_executed`, `rollback_required`, `rollback_executed`, irreversible
warning, and next action in text and JSON without relaxing the existing gates.
V1.14.5 adds the OS service-manager abstraction baseline: `eva-lifecycle`
defines the service-manager adapter trait, fake/Windows Service/systemd/launchd
kinds, typed handoff/rollback reports, and a fake adapter for local handoff and
rollback tests. `eva-config` parses explicit top-level `service_manager`
configuration, and `release check` reports `REL-SERVICE-MANAGER-ABSTRACTION-001`
while platform adapters remain blocked until V1.14.6 test environments exist.
V1.15.1 starts the hardware-realization wave with platform-aware OS permission
diagnostics. `eva-hardware` exposes `PlatformOsPermissionProvider` and enriches
`OsPermissionCheck` with OS, user, permission source, a safe logical device
locator, remediation, and `raw_device_path_exposed:false` evidence.
`HardwareLifecycleCoordinator` denies missing permission before lease claim or
driver start, `hardware bind` reports permission evidence, and `SEC-HW-001`
records the boundary while real USB/serial/BLE/socket/vendor SDK I/O remains
blocked by fixture-backed driver work.
V1.15.4 adds the daemon hotplug subscriber baseline. `eva-hardware` can run a
manifest-snapshot watcher that emits typed logical insert/remove/reconnect/fail
events without exposing raw handles, persists `hardware-hotplug.state`, and
keeps watcher-crash lease cleanup testable. `daemon start` writes those events
to the durable EventBus and reports additive `hardware_hotplug` evidence while
real OS hotplug notifications remain future fixture/platform work.
V1.15.5 adds the hardware safety release gate. `release check` now reports the
required `REL-HARDWARE-SAFETY-001` gate with simulator parity, permission
denial, lease cleanup, and daemon hotplug smoke evidence. Alpha accepts
simulator-only evidence; production releases still need real or virtual hardware
fixture evidence before claiming real USB/serial/BLE/socket/vendor SDK I/O.
V1.15.6 adds the durable memory/knowledge maintenance baseline. Filesystem
stores now acquire `index.lock` for reads and writes, memory TTL GC removes
expired files and writes `memory-gc.checkpoint`, knowledge rebuild writes
`knowledge-rebuild.checkpoint`, stale started checkpoints can be recovered, and
`daemon start` reports additive `memory_maintenance` evidence. Provider-backed
retrieval, policy-driven redaction audit, and a long-lived background scheduler
remained later slices at this checkpoint.
V1.15.7 adds the supervised knowledge retrieval provider execution baseline.
`eva-memory` can now pass the runtime policy gate, invoke a specific provider
through `CapabilityHostApi::invoke_with_provider`, accept only a line-based hex
retrieval schema, redact sensitive provider results before indexing, and return
`ExternalKnowledgeRetrievalReport` source audit evidence. Provider failures,
timeouts, and invalid schemas do not pollute the knowledge index. V1.15.8 adds
`memory_policy.redaction` typed policy parsing, policy-driven memory/knowledge
redaction, memory write/read/search/context audit actions, memory operation and
redaction metrics, CLI `memory context` JSONL evidence, and the
`memory_redaction_audit_v1.15.8` runtime marker. Production retrieval
scheduling and a long-lived background scheduler remain later slices. V1.16.1
then wires the existing JSONL best-effort observability
pipeline into daemon recovery/control, task lifecycle, scheduler retry,
provider supervision, and restore apply/rollback paths, adding the
`runtime_audit_sink_wiring_v1.16.1` runtime marker. V1.16.2 adds the
`tracing_subscriber_bridge_v1.16.2` runtime marker and bridges tracing
spans/events into existing TraceFields/AuditEvent plus JSONL/dev-console sinks,
while V1.16.3 adds the `opentelemetry_sdk_exporter_v1.16.3` runtime marker,
SDK-based OTLP HTTP/protobuf trace and metrics exporter smoke, collector
degraded reporting, and bounded metric label cardinality. V1.16.4 adds the
`observability_retention_policy_v1.16.4` runtime marker, JSONL/durable-audit
retention and rotation policy, corrupt-record handling, and a database policy
kind boundary. A real database sink remains later work.
V1.11.5 has started after the CLI command module split: V1.11.5.1 adds the
`emit` command, which constructs typed `eva-core::Event` values and publishes
them to either in-memory EventBus or a V1.6 durable backend EventBus while
returning stable text/JSON receipts. V1.11.5.2 adds `agent status`,
`agent drain`, and `agent reload` as CLI evidence boundaries over
ProjectConfig manifests, `AgentRuntime`, `AgentLifecycle`, `DrainCoordinator`,
and `GenerationController`; these commands report lifecycle/drain/reload
evidence with `mutation_executed:false` and do not restart a daemon or mutate a
live scheduler. V1.11.5.3 adds `capability list`, `capability probe`, and
`capability call`; these commands reuse `CapabilityRegistry`, provider
selection plans, `CapabilityPermissionGate`, `RuntimePolicyGate`,
`CapabilityRouter`, and `AdapterBackedCapabilityHost`. `capability call`
defaults to dry-run and only invokes after `--confirm <request-id>`, while
manifest/provider allowlist violations return policy errors. V1.11.5-alpha is
now the current source alpha checkpoint for this command-completion slice.
The detailed Chinese plan now adds the next implementation wave for the
remaining gaps: V1.12 daemon runtime and scheduler apply, V1.13 provider
supervision and full MCP, V1.14 destructive restore and production
service-manager handoff, V1.15 real hardware plus background memory/knowledge,
V1.16 production observability plus release upload, and V1.17 CLI/operator UX
closure. Each future slice has a fine-grained progress table with prerequisites,
implementation logic, validation, and status.

## Version Slices

| Version | Theme | Feature Iterations | Target Result |
| --- | --- | ---: | --- |
| V1.6 | Durable runtime/storage | 5 | EventBus, task, audit, and artifact paths become recoverable after process restart. |
| V1.7 | Real Lua VM and hot reload | 4 | Lua Agents execute real script logic with resource limits, host APIs, shadow-load health checks, generation route gates, drain evidence, and rollback audit evidence. |
| V1.8 | External providers, MCP, and Skill runner | 5 | Adapter/MCP/Skill move from probe/envelope paths to controlled real process or protocol execution. |
| V1.9 | Policy, Discovery, Memory, and Observability hardening | 5 | Config, permission, discovery, memory, and observability become durable, auditable runtime gates. |
| V1.10 | Real hardware and high-risk apply paths | 5 | Hardware, restore, and upgrade paths move from plans to confirmable, rollback-aware real execution. |
| V1.11 | Distribution and CLI convergence | 5 | Signed artifacts, installers, external scanners, and CLI command modules form the release closure. |
| V1.12 | Daemon runtime and scheduler apply | 6 | Runtime, task recovery, and agent drain/reload become daemon-backed, controllable, and recoverable. |
| V1.13 | Provider supervision and full MCP | 8 | Providers and MCP run through supervised process/session, concurrency, streaming, and recovery boundaries. |
| V1.14 | Destructive restore and service-manager handoff | 8 | Restore performs gated file mutation, and upgrade uses production service-manager blue-green handoff. |
| V1.15 | Real hardware and background memory service | 8 | Real drivers, OS permissions, hotplug subscribers, memory compaction, and retrieval providers land. |
| V1.16 | Production observability and release upload | 7 | Runtime/provider/task telemetry, OTel/metrics export, signing, attestation, and package uploads land. |
| V1.17 | CLI closure and V1.x release readiness | 6 | Operator UX, JSON contract diffs, docs sync, and final release gates close the V1.x plan. |

## Implementation Guardrails

- Keep V1.0-V1.5 public CLI commands, JSON envelopes, and exit codes compatible by default.
- Add regression tests before widening side effects from dry-run, envelope, or plan-only paths.
- Keep high-risk operations plan-first, including restore, upgrade, hardware bind, release pointer mutation, and permission expansion.
- Do not give Lua, Agents, or Discovery direct provider, device, file, socket, or process handles.
- Update the detailed Chinese plan, incomplete feature inventory, user manual, release docs, and i18n manifest whenever a real execution path lands.

Use the Chinese detailed plan for per-version progress tables, prerequisites, concrete implementation logic, module ownership, and acceptance gates.
