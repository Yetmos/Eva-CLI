# V1.x Real Runtime Implementation Plan

This page is the English entry for the V1.x real runtime implementation plan. The full-detail source remains the Simplified Chinese document: [V1.x真实运行时能力补齐实施计划.md](../../zh-CN/planning/V1.x真实运行时能力补齐实施计划.md).

The plan is based on the [V1.x Incomplete Feature Inventory](v1.x-incomplete-feature-inventory.md). It turns the V1.5.1 gaps that still sit at contract, diagnostic, plan-first, in-memory, simulated, or envelope boundaries into a sequence of small implementation releases.

Current progress: V1.6 is complete through V1.6.5-alpha. It has landed the
durable backend manifest, layout, read-only verification, migration-lock
baseline, filesystem EventLog, durable EventBus dead-letter redrive, durable
task snapshots through `--durable-backend <path>`, `FileSystemAuditSink`
persistence under `audit/` with trace-id lookup, filesystem artifact metadata
hardening with content type, retention, legacy compatibility, stable
corrupt-metadata errors, runtime recovery scanning, event redrive checkpoints,
durable `runtime.recovered` audit records, clean-start/restart/corrupt-store
smoke, `inspect durable` backend diagnostics, and the
`REL-DURABLE-RECOVERY-001` and `REL-DURABLE-DIAGNOSTICS-001` release gates.
V1.7.4-alpha has now advanced the real Lua VM line by adding a restricted
`mlua`/Lua 5.4 VM adapter, real `on_event(ctx)` execution, stable syntax/runtime
error mapping, compatibility fallback for legacy static scripts, read-only Lua
context tables, `ctx.host.log/audit`, `ctx.tools.call`, Lua wall-clock timeout,
instruction budget, cancellation token, memory budget, shadow-load health
checks, scheduler generation route gating, generation drain evidence, rollback
audit evidence, and the
`REL-LUA-VM-EXECUTION-001`, `REL-LUA-HOST-BINDINGS-001`, plus
`REL-LUA-RESOURCE-LIMITS-001` and `REL-LUA-HOT-RELOAD-001` release gates.
Scheduler backoff dispatch, daemon-driven hot-reload orchestration, provider
process recovery, and complete runtime state restoration remain planned
follow-up slices.
V1.8 development has started after that alpha checkpoint: V1.8.1 wires
manifest-derived stdio/http adapter runners into `AdapterRuntime`, and V1.8.2
adds controlled MCP JSON-RPC stdio tool calls with allowlist, timeout,
output-limit, and protocol-error gates. V1.8.3 adds an MCP session registry,
health/shutdown/orphan cleanup, stream-abort boundary, explicit-tool server
gate, and stable MCP audit actions. V1.8.4 adds a controlled Skill workflow
runner with manifest input-schema gates, isolated working directories,
manifest allowlisted process execution, a managed `codex_skill` runner for the
sample adapter, filesystem artifact evidence, timeout/failure coverage, and
credential redaction. Real OS process supervision, provider concurrency/rate
limits, richer Skill output-schema validation, and production policy domains
remain planned follow-up slices.
V1.11.5 has started after the CLI command module split: V1.11.5.1 adds the
`emit` command, which constructs typed `eva-core::Event` values and publishes
them to either in-memory EventBus or a V1.6 durable backend EventBus while
returning stable text/JSON receipts. V1.11.5.2 adds `agent status`,
`agent drain`, and `agent reload` as CLI evidence boundaries over
ProjectConfig manifests, `AgentRuntime`, `AgentLifecycle`, `DrainCoordinator`,
and `GenerationController`; these commands report lifecycle/drain/reload
evidence with `mutation_executed:false` and do not restart a daemon or mutate a
live scheduler. `capability` operator commands remain the next V1.11.5 slice.

## Version Slices

| Version | Theme | Feature Iterations | Target Result |
| --- | --- | ---: | --- |
| V1.6 | Durable runtime/storage | 5 | EventBus, task, audit, and artifact paths become recoverable after process restart. |
| V1.7 | Real Lua VM and hot reload | 4 | Lua Agents execute real script logic with resource limits, host APIs, shadow-load health checks, generation route gates, drain evidence, and rollback audit evidence. |
| V1.8 | External providers, MCP, and Skill runner | 5 | Adapter/MCP/Skill move from probe/envelope paths to controlled real process or protocol execution. |
| V1.9 | Policy, Discovery, Memory, and Observability hardening | 5 | Config, permission, discovery, memory, and observability become durable, auditable runtime gates. |
| V1.10 | Real hardware and high-risk apply paths | 5 | Hardware, restore, and upgrade paths move from plans to confirmable, rollback-aware real execution. |
| V1.11 | Distribution and CLI convergence | 5 | Signed artifacts, installers, external scanners, and CLI command modules form the release closure. |

## Implementation Guardrails

- Keep V1.0-V1.5 public CLI commands, JSON envelopes, and exit codes compatible by default.
- Add regression tests before widening side effects from dry-run, envelope, or plan-only paths.
- Keep high-risk operations plan-first, including restore, upgrade, hardware bind, release pointer mutation, and permission expansion.
- Do not give Lua, Agents, or Discovery direct provider, device, file, socket, or process handles.
- Update the detailed Chinese plan, incomplete feature inventory, user manual, release docs, and i18n manifest whenever a real execution path lands.

Use the Chinese detailed plan for per-version progress tables, prerequisites, concrete implementation logic, module ownership, and acceptance gates.
