# V1.x Incomplete Feature Inventory

Updated: 2026-07-07

This document records Eva-CLI `V1.6.2-alpha` features that already have partial implementation but have not yet entered complete real execution paths. It complements the [Full Implementation Plan](full-implementation-plan.md) by separating implemented contracts, diagnostics, plans, and controlled envelopes from runtime capabilities that still need to be completed.

Current conclusion: V1.6.2-alpha has a buildable workspace, stable CLI output, an in-memory basic runtime, external capability diagnostics, memory context, hardware binding plans, backup/lifecycle plans, release-hardening gates, the first durable backend layout/migration baseline, and durable EventBus redrive. V1.6.3 has started wiring task snapshots into the durable backend `tasks/` layout. It is still an alpha checkpoint, not a complete runtime distribution with durable audit recovery, real providers, real hardware, real apply gates, and signed installers.

## Status Terms

| Status | Meaning |
| --- | --- |
| Boundary exists | Types, traits, commands, diagnostics, or plan-first outputs exist and can serve as the basis for later implementation. |
| Partially implemented | The feature can run under controlled conditions, but only through in-memory, mock, simulated, dry-run, or envelope paths. |
| Incomplete | Real side effects, persistence, process management, external protocol execution, OS permissions, signed release artifacts, or complete policy gates are not yet wired. |
| Next step | Upgrade the existing boundary into an auditable, rollback-aware, testable real execution path. |

## Suggested Upgrade Order

| Stage | Slice | Goal | Exit Criteria |
| --- | --- | --- | --- |
| P0-A | Durable storage + runtime crash recovery | Give EventBus, task, audit, and artifact paths recoverable backends first | Task/event/audit can still be queried after restart; `cargo test --workspace` and durable integration tests pass |
| P0-B | Real Lua VM + host API | Upgrade controlled static parsing into real Lua execution | Lua sandbox, timeout, memory limit, and host API permission tests pass |
| P0-C | Adapter/MCP real providers | Enable stdio/http/MCP providers while preserving manifest, policy, audit, and timeout boundaries | Unauthorized providers do not start; timeout/rate-limit/output-limit/redaction tests pass |
| P0-D | Restore/upgrade apply gate | Move from plan-only to confirmable, rollback-aware real apply | Signed artifacts, backup evidence, apply lock, health check, and rollback tests pass |
| P1-A | Real hardware driver path | Lock simulator regression and OS permission model before real device access | Real drivers do not expose raw handles; hardware simulator is repeatable in CI |
| P1-B | Policy/schema/observability hardening | Upgrade config, permissions, and audit from base contracts into runtime gates | Schema validator, policy domain parser, and persistent audit sink pass integration tests |
| P1-C | Release artifact distribution | Add signing, installers, package-manager packages, and provenance | release check verifies artifact evidence and blocks missing items |

## Maintenance Rules

| Rule | Description |
| --- | --- |
| Test before widening side effects | Before upgrading from diagnostic/envelope/plan to real execution, add failure cases and regression tests first. |
| Keep plan-first behavior | High-risk paths such as restore, upgrade, hardware, release pointer changes, and permission expansion must produce plans and risks before mutation. |
| Do not bypass boundaries | Lua, Agents, and Discovery must not receive direct provider, device, file, socket, or process handles. |
| Keep docs aligned with implementation | When any real execution path lands, update this inventory, the implementation plan, the user manual, and release docs. |
