# V1.x Incomplete Feature Inventory

Updated: 2026-07-10

This page is the English summary for the current V1.x implementation gap,
blocked-feature, and known-risk inventory. The detailed source of truth is the
Simplified Chinese document:
[V1.x 未完整实现功能清单](../../zh-CN/planning/V1.x未完整实现功能清单.md).

Eva-CLI `V1.11.5-alpha` has a buildable workspace, stable CLI envelopes,
in-memory runtime evidence, durable storage/recovery baselines, restricted Lua
execution, controlled Adapter/MCP/Skill paths, hardware and lifecycle gates,
release evidence gates, and the `emit`, `agent`, and `capability` runtime CLI
commands. V1.12.6 also lets `agent drain/reload` mutate daemon-side
`agent-control.state` when a foreground daemon is running and adds the
`REL-DAEMON-RUNTIME-001` release readiness gate. V1.13.1 adds the first provider
supervisor slot and in-memory provider process table for stdio/http/MCP/Skill
execution evidence. V1.13.2 binds provider credential env/header/session tokens
to the provider session/request/capability scope, rejects cross-provider scope
reuse, and keeps token redaction in stdout/stderr/body/artifact evidence, while
preserving no-daemon evidence-only fallbacks. V1.13.3 adds manifest-derived
provider concurrency, rate-limit, and circuit-breaker admission gates with
retryable backoff hints, circuit-open health, and half-open probes. V1.13.4
adds bounded provider stream summaries, truncation, redacted artifact sinks, and
chunked HTTP response reads. V1.13.5 adds durable provider process snapshots,
daemon-start provider execution-state recovery, interrupted stale active
sessions, and explicit retryable restart backoff evidence. V1.13.6 adds bounded
MCP `http://` JSON-RPC transport/auth evidence, including manifest endpoints
and headers, provider session headers, allowlist-before-RPC, fake HTTP server
e2e, and missing-auth policy errors. V1.13.7 adds an MCP compatibility matrix
fixture and required `REL-MCP-COMPAT-001` release gate for stdio/HTTP transport,
tool schema, stream lifecycle abort/cleanup, dangling-session, and explicit
server-surface evidence. V1.13.8 adds required
`REL-PROVIDER-SUPERVISION-001` release gate for the current controlled provider
supervision baseline: supervisor slots, process-table evidence, credential
scope, admission limits, stream artifacts, daemon recovery, and MCP
compatibility readiness. It is still an alpha checkpoint, not a production
daemon distribution, full OS provider supervisor, MCP production streaming/TLS
or external-server compatibility certification, or OS credential vault.
V1.14.1 adds the plan-only restore staged mutation planner: restore apply plans
can describe copy/delete/replace steps, affected paths, a deterministic
preflight hash, preview text, and a rollback manifest while still reporting
`mutation_executed:false`.
V1.14.2 adds the gated staged file mutation engine. When confirmation,
pre-restore evidence, policy approval, lock, health, and staged mutation steps
are all present, `restore apply` can execute copy/delete/replace file mutation,
write a transaction log, report `mutation_executed:true`, or stop with
`rollback_required`.
V1.14.3 adds rollback apply. `eva restore rollback` reuses the apply plan,
confirmation, artifact evidence, policy allow, rollback lock, and health gate,
then accepts only a `rollback_required` transaction log with a matching staged
preflight hash. It restores committed steps in reverse order from pre-restore
archive entries, rejects current target digest drift, writes
`{plan_id}.restore.rollback.txn`, and returns `rollback_executed:true` only after
reverse mutation commits.
V1.14.4 adds restore operator confirmation output. `restore apply --dry-run`,
`restore apply`, and `restore rollback` now expose an additive
`operator_confirmation` object in JSON and matching text fields for the confirm
token, plan id, target root, affected count, state flags, irreversible warning,
and next action. This clarifies operator state without replacing confirmation,
policy, lock, health, transaction-log, or digest gates.
V1.14.5 adds the service-manager abstraction baseline. `eva-lifecycle` now has a
service-manager adapter trait, fake/Windows Service/systemd/launchd kinds,
typed handoff/rollback reports, and a fake adapter for local handoff and
rollback tests. `eva-config` parses explicit `service_manager` configuration,
and `release check` exposes `REL-SERVICE-MANAGER-ABSTRACTION-001`. Platform
adapters and production blue-green handoff remain later blocked work.
V1.15.1 adds the hardware OS permission provider boundary.
`PlatformOsPermissionProvider` reports OS, user, permission source, a safe
logical device locator, remediation, and `raw_device_path_exposed:false`.
Permission denial now blocks before lease claim or driver start, and
`hardware bind` exposes the permission evidence without opening a real device.
Real USB/serial/BLE/socket/vendor SDK I/O still depends on fixture-backed driver
work.
V1.15.4 adds a daemon hotplug subscriber baseline. The subscriber reads the
manifest snapshot, emits typed logical insert/remove/reconnect/fail events to
the durable EventBus, persists `hardware-hotplug.state`, reports
`raw_handles_exposed:false`, and releases leases on watcher crash. Real OS
hotplug notifications and real device I/O remain future driver/platform work.
V1.15.5 adds the required hardware safety release gate. `release check` now
reports `REL-HARDWARE-SAFETY-001` with simulator parity, permission denial,
lease cleanup, and daemon hotplug smoke evidence. This is an alpha
simulator-only gate; production releases still require real or virtual hardware
fixture evidence before claiming real device I/O.
V1.15.6 adds the durable memory/knowledge maintenance baseline. Memory and
knowledge stores now use filesystem `index.lock` guards, memory TTL GC writes
`memory-gc.checkpoint`, knowledge rebuild writes `knowledge-rebuild.checkpoint`,
stale started checkpoints can be recovered, and `daemon start` exposes
`memory_maintenance` evidence. Provider-backed retrieval, policy-driven
redaction audit, and a long-lived background scheduler remained future work at
that checkpoint.
V1.15.7 adds the supervised knowledge retrieval provider execution baseline.
`eva-memory` now invokes a selected provider through `CapabilityHostApi`,
accepts only a retrieval schema, redacts sensitive returned knowledge before
indexing, and exposes source audit evidence. Provider failure, timeout, policy
denial, and schema rejection do not pollute the knowledge index. V1.15.8 adds
`memory_policy.redaction` typed parsing, policy-driven memory/knowledge
redaction, memory write/read/search/context audit actions, memory operation and
redaction metrics, CLI `memory context` JSONL evidence, and the
`memory_redaction_audit_v1.15.8` runtime marker. Production retrieval scheduling
and a long-lived background scheduler remain future work.
V1.16.1 adds runtime/provider/task audit sink wiring over the existing JSONL
best-effort pipeline. Daemon recovery/control, submit/cancel task lifecycle,
scheduler retry ticks, provider supervisor invoke/admission/policy/transport
failures, and successful non-dry-run restore apply/rollback paths now write
audit, metric, and span JSONL records. V1.16.2 adds a `tracing-subscriber`
bridge that maps tracing spans/events into existing TraceFields, AuditEvent,
and OTel-style JSONL span records, with JSONL/dev-console sinks and secret
redaction. This still does not add an OpenTelemetry SDK/exporter, production
metrics exporter, retention/rotation, or db sink. V1.16.3 then adds an
SDK-based OTLP HTTP/protobuf trace and metrics exporter smoke with endpoint,
auth, batch, timeout, drop-policy, collector-degraded reporting, and bounded
metric label cardinality. V1.16.4 adds JSONL/durable-audit retention,
rotation, max-size, and corrupt-record policy plus a database policy kind
boundary. A real database sink and production retention scheduler remain future
work. V1.17.1 moves `run --example basic` parser/runtime glue/task snapshot and
output code into `run/run_cmd.rs`, while V1.17.2 fixes operator execution-state
fields: capability calls expose `invocation_executed` plus
`mutation_executed:false`, and restore, upgrade, and hardware outputs expose
top-level `mutation_executed` while preserving existing compatibility fields.
V1.17.3 adds text-only operator summaries for restore, upgrade, and hardware
high-risk paths, including plan/confirm, target, final state, rollback path, and
risk lines. V1.17.4 adds golden subset JSON fixtures under
`contracts/cli-json/`, `scripts/validate-cli-json-contracts.ps1`, the
`json_contract_diff_suite_v1.17.4` runtime marker, and the required
`REL-JSON-CONTRACT-001` release gate. Public JSON contract validation now blocks
removed or renamed fields while allowing additive fields. Production release
upload remains blocked by credentials and repository permissions.

## Blocked Features

| Feature | Why Blocked | Current Evidence | Unblock Condition |
| --- | --- | --- | --- |
| Daemonized supervisor handoff | Current handoff is local evidence plus V1.14.5 service-manager abstraction, not a production service manager | `release check` keeps `SEC-LIFE-001` as a warning gate; `REL-SERVICE-MANAGER-ABSTRACTION-001` only proves the abstraction layer | Implement OS service-manager adapter and production rollback |
| Hardware raw I/O | Sample hardware manifest is disabled and CLI/daemon never grant raw handles | `hardware bind --apply` reports `status:"blocked"`, `handle_granted:false`, `permission.raw_device_path_exposed:false`, and remediation before lease claim; `daemon start` reports `hardware_hotplug.raw_handles_exposed:false` and writes `hardware-hotplug.state`; `release check` reports `REL-HARDWARE-SAFETY-001` alpha simulator-only evidence | Add real drivers, real OS hotplug watcher, and real or virtual hardware fixtures |
| Unauthorized providers and MCP tools | Discovery and manifest visibility are not execution authority | provider allowlists and MCP allowlists block unauthorized calls before invoke | Add supervision and policy wiring while preserving allowlist/audit gates |
| Production release upload | Evidence gates validate readiness but do not hold production credentials | release checks verify evidence and do not upload packages | Add signing/attestation credentials and package repository publication |

## Known Defects And Risks

No verified compile failure or command failure was found during this document
refresh. `cargo run -- --version` and `cargo run -- release check --output json`
passed. The current known issues are capability or product risks rather than
reproduced build failures.

| Type | Item | Risk | Recommendation |
| --- | --- | --- | --- |
| UX risk | `capability call` | Confirmed invokes can execute while `mutation_executed:false` remains correct for destructive mutation semantics; V1.17.2 now shows both execution fields | Keep `invocation_executed` and `mutation_executed` together in operator-facing output and docs |
| UX risk | `restore apply` | Legacy no-step plans can still be gated/no-op, while staged plans can now apply real file mutation and V1.17.3 text output exposes final state, rollback path, and risk lines | Keep `operator_confirmation`, `mutation_planned`, `mutation_executed`, `mutation_apply.status`, `rollback_required`, `rollback_executed`, `final_state`, and `rollback_path` explicit in operator output |
| Compatibility risk | public CLI JSON | V1.17.4 golden subset fixtures cover version, run basic, capability list, hardware bind, restore plan, upgrade check, and release check | Run `scripts/validate-cli-json-contracts.ps1`; treat removed or renamed fields as breaking unless a compatibility window is documented |
