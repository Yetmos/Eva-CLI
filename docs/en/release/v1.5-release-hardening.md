# Eva-CLI V1.5 Release Hardening

Date: 2026-07-04

V1.5 turns the remaining 1.x release-readiness work into executable checks.
The release is still a source-release checkpoint, but it now has a public
hardening surface for cross-platform readiness, security review, performance
budgets, migration steps, compatibility policy, and release notes.

## Supported Commands

```powershell
cargo run -- release check --output json
cargo run -- release check --target windows --output json
cargo run -- release security --output json
cargo run -- release perf --output json
cargo run -- release migration --output json
```

## Release Gates

`release check` aggregates these gate groups:

| Gate Group | Scope | Blocking Rule |
| --- | --- | --- |
| Cross-platform | Windows, Linux, macOS CI smoke commands and shell/path assumptions | Any required platform gate blocked prevents release readiness. |
| Stability | task status/logs/cancel, timeout dead-letter replay, restore planning, upgrade rollback planning | Recovery paths must remain auditable. |
| Documentation | V1.5 release docs, migration guide, compatibility policy, module README updates | Public docs must match the executable command surface. |
| Security | policy, Lua sandbox, secret redaction, MCP allowlist, hardware handles, lifecycle apply risk | High/critical blocked findings prevent readiness. |
| Performance | EventBus, Scheduler, Adapter probe, memory context, backup, release check budgets | Any over-budget required budget blocks release readiness. |
| Migration | V1.4 -> V1.5 migration guide and compatibility policy | Breaking changes require explicit documentation and tests. |

## Security Review

`release security` reports the security findings tracked by `eva-release`:

- policy narrowing rejects permission expansion;
- Lua host receives controlled context snapshots instead of host handles;
- backup manifests preserve redaction metadata for sensitive entries;
- MCP probes are allowlisted and do not start external servers;
- hardware discovery never grants raw I/O handles from the CLI;
- restore and upgrade apply paths remain warning-tracked until signed artifacts
  and explicit apply authorization exist.

## Performance Baseline

`release perf` defines source-release budgets for the current in-memory
implementation. The budgets are release-smoke contracts rather than production
benchmarks. They cover:

- `eventbus.publish`
- `scheduler.fanout`
- `adapter.probe`
- `memory.context`
- `backup.create`
- `release.check`

## Non-Goals

V1.5 does not add packaged installers, signed release artifacts, real MCP
process execution, real hardware I/O, destructive restore, or real Supervisor
process management. Those remain future work and must not be inferred from the
release-hardening command group.
