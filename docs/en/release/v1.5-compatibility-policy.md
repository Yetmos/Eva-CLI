# Eva-CLI V1.5 Compatibility Policy

Date: 2026-07-04

V1.5 makes the source-release compatibility contract explicit. The goal is to
protect scripts and CI jobs that already depend on the V1.0-V1.4 command
surface while adding release-hardening commands.

## Stable Contracts

| Contract | V1.5 Policy |
| --- | --- |
| JSON envelope | Success output keeps `ok`, `command`, `exit_code`, `data`, and `trace`; error output keeps `ok`, `command`, `exit_code`, `error`, and `trace`. |
| Exit codes | `0`, `1`, `2`, `3`, `4`, `5`, and `64` keep their documented meanings. |
| Command surface | V1.5 only adds `release check/security/perf/migration`; previous commands remain available. |
| Config samples | Existing sample manifests remain loadable without migration. |
| Plan-first operations | hardware bind, restore, and upgrade checks remain non-destructive unless a future version documents an explicit apply gate. |
| Deprecation | Breaking CLI or manifest changes require one documented release window. |

## Public Command Families

- `eva --version`
- `eva version --output json`
- `eva doctor`, `eva config validate`, `eva inspect`
- `eva run --example basic`
- `eva task status/logs/cancel`
- `eva adapter`, `eva mcp`, `eva skill`, `eva discovery`
- `eva memory context`
- `eva hardware list/probe/bind`
- `eva backup create`, `eva snapshot create`, `eva restore plan`, `eva upgrade check`
- `eva release check/security/perf/migration`

## Compatibility Limits

This policy covers the source checkout, CLI commands, JSON envelopes, and
documented sample configuration. It does not promise packaged installer
compatibility, signed artifact layout, production benchmark numbers, or durable
storage schema stability.
