# Eva-CLI V1.0 Quickstart

This quickstart is the supported Eva-CLI V1.0 core path. It proves that a new checkout can build the CLI, validate configuration, inspect the runtime boundary, run the basic Agent loop, and diagnose the local task report.

## Prerequisites

- Git.
- A stable Rust toolchain with `cargo`, `rustfmt`, and `clippy` available on `PATH`.
- PowerShell 7+ only if you want to run the website i18n scripts locally. The Rust quickstart works from PowerShell, Bash, or any shell that can run `cargo`.

No external services, daemons, databases, Lua VM packages, MCP servers, hardware drivers, or API keys are required for V1.0 core.

## Install From Source

```powershell
git clone https://github.com/Yetmos/Eva-CLI.git
cd Eva-CLI
cargo build --release
```

The built binary is available at `target/release/eva` on Linux/macOS and `target/release/eva.exe` on Windows. During development, the examples below use `cargo run -- ...` so they do not require copying the binary into `PATH`.

## Verify The Release Surface

```powershell
cargo run -- --version
cargo run -- version --output json
cargo run -- doctor --output json
cargo run -- config validate --output json
cargo run -- inspect runtime --output json
```

Expected signals:

- `eva --version` prints `eva 1.0.0` and `release: V1.0 core`.
- `eva version --output json` returns the standard success envelope with `runtime_mode` set to `in_memory_v1.0`.
- `doctor` reports configuration roots, schema files, runtime boundaries, and Lua host boundary checks.
- `config validate` loads `config/eva.yaml`, manifests, policies, routes, and schema paths.
- `inspect runtime` reports the no-op inspection boundary used by diagnostics.

## Run The Basic Loop

```powershell
cargo run -- run --example basic --task-id req-quickstart-v10 --output json
```

The V1.0 basic loop runs synchronously:

1. CLI loads `examples/basic/config/eva.yaml`.
2. Runtime builds the `in_memory_v1.0` core service table.
3. EventBus accepts `/input/user` and appends the event.
4. Scheduler routes the event to `root-agent`.
5. AgentRuntime applies timeout/cancel/retry controls.
6. Lua host validates the controlled `on_event` table contract.
7. Capability router invokes the builtin `config.lint` capability.
8. CLI writes `.eva/tasks/req-quickstart-v10.task` and `.eva/tasks/latest-basic.task`.

## Inspect Task Diagnostics

```powershell
cargo run -- task status --task req-quickstart-v10 --output json
cargo run -- task logs --task req-quickstart-v10 --output json
cargo run -- task cancel --task req-quickstart-v10 --reason quickstart-check --output json
```

`task cancel` on an already completed task records the cancel request for auditability but does not change the terminal `completed` status.

## Exercise Failure Diagnostics

```powershell
cargo run -- run --example basic --task-id req-quickstart-timeout --timeout-ms 0 --replay-dead-letters --output json
cargo run -- run --example basic --task-id req-quickstart-cancel --cancel --output json
```

The timeout command should produce a `timed_out` task with one dead letter and one replay summary. The cancel command should produce a `cancelled` task with a dead-letter diagnostic and no Lua/capability invocation.

## Release Gates

Run these commands before changing V1.0 core behavior:

```powershell
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
./scripts/build-site-i18n.ps1
./scripts/validate-i18n.ps1
```

The GitHub Actions workflow `.github/workflows/ci.yml` runs the Rust gates across Ubuntu, Windows, and macOS, then runs the website/i18n gates on Ubuntu.

## Where State Is Written

V1.0 writes only local task diagnostic files under `.eva/tasks`. The directory is ignored by Git and is not a stable database API. Delete `.eva/tasks` when you want to clear local quickstart state.
