# Eva-CLI V1.5 GitHub Release Plan

Date: 2026-07-04
Release tag: `v1.5.0`
Version owner: GitHub tags and GitHub Releases

## Goal

Publish Eva-CLI V1.5.0 as a GitHub-managed source release. The repository has
already reached the V1.5 release-hardening checkpoint, so the release plan keeps
version authority in GitHub:

- the immutable version record is the annotated Git tag `v1.5.0`;
- the public release page is the GitHub Release created from that tag;
- release evidence is produced by GitHub Actions after the tag is pushed;
- V1.5 remains a source release, not a packaged binary distribution.

## Scope

Included:

- tag-driven GitHub Release workflow;
- Rust workspace verification on Windows, Linux, and macOS;
- website and documentation generation/validation before release publication;
- V1.5 release-gate evidence from `eva release check/security/perf/migration`;
- GitHub source archives managed by GitHub.

Not included:

- signed installers or packaged binary artifacts;
- destructive `restore apply`;
- real Supervisor process handoff;
- real MCP server startup;
- hardware raw I/O;
- production benchmark certification.

## Release Preconditions

Before pushing `v1.5.0`, the release commit on `main` must contain:

1. `Cargo.toml` package/workspace version `1.5.0`.
2. V1.5 release notes, migration guide, compatibility policy, and release
   hardening docs.
3. The GitHub Release workflow at `.github/workflows/release.yml`.
4. A clean local working tree.
5. Passing local checks:
   - `cargo fmt --check`
   - `cargo clippy --workspace --all-targets -- -D warnings`
   - `cargo test --workspace`
   - `./scripts/build-site-i18n.ps1`
   - `./scripts/validate-i18n.ps1`
   - `cargo run -- release check --output json`
   - `cargo run -- release security --output json`
   - `cargo run -- release perf --output json`
   - `cargo run -- release migration --output json`

## Execution Plan

1. Commit and push documentation/website updates.
2. Commit and push this release plan plus `.github/workflows/release.yml`.
3. Create an annotated tag:

```powershell
git tag -a v1.5.0 -m "Eva-CLI V1.5.0"
```

4. Push the tag:

```powershell
git push origin v1.5.0
```

5. Let GitHub Actions run `GitHub Release`.
6. GitHub Actions verifies:
   - tag format and Cargo version match;
   - Rust format, clippy, tests, and V1.5 smoke commands;
   - localized website build and i18n validation;
   - release check/security/perf/migration evidence.
7. GitHub Actions creates or updates the GitHub Release for `v1.5.0`.

## Post-Release Verification

After pushing the tag, verify from git:

```powershell
git ls-remote --tags origin v1.5.0
```

Then verify in GitHub Actions that the `GitHub Release` workflow completed.
When `gh` is unavailable locally, the Git remote tag is the local proof that
version ownership has moved to GitHub; the Release page is created by the
remote workflow.

## Rollback And Repair

If the release workflow fails before a GitHub Release is created:

1. Fix the failing commit on `main`.
2. Delete the local tag:

```powershell
git tag -d v1.5.0
```

3. Delete the remote tag only if no public release has been published yet:

```powershell
git push origin :refs/tags/v1.5.0
```

4. Recreate and push `v1.5.0` from the repaired commit.

If a GitHub Release already exists, prefer creating a patch release such as
`v1.5.1` instead of rewriting public history.

## Responsibility Split

| Responsibility | Owner |
| --- | --- |
| Semantic version value | `Cargo.toml` and release docs |
| Version identity | Git tag `v1.5.0` |
| Public release page | GitHub Release workflow |
| Release evidence | GitHub Actions logs and `release-evidence-*` artifact |
| Source archives | GitHub generated archives |
| Installers/signatures | Future release scope |
