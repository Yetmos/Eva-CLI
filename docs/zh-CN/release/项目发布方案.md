# Eva-CLI 项目发布方案

日期：2026-07-06
范围：项目级发布流程、CI/DI 门禁、GitHub Packages，以及 Windows、macOS、Linux 支持

本文定义 Eva-CLI 的项目级发布方案。它位于 V1.5 GitHub 发版计划等版本专项文档之上，描述每次发布都应复用的准备、验证、发布和修复流程。

本文中的 CI/DI 含义如下：

- CI：持续集成，用于 pull request、主分支推送和手动触发的构建验证。
- DI：交付/部署集成，用于 GitHub Release 发布、GitHub Packages 发布、发布证据采集和文档站点部署。

## 发布目标

发布流程必须提供：

- 通过 Git tag 和 GitHub Release 可重复发布源码版本；
- 发布前在 Windows、macOS、Linux 三个平台完成验证；
- 可在发布后审计的 release evidence；
- 公开交付前完成文档和网站校验；
- 对包含 package 支持的 release tag，通过 GHCR 发布受控 GitHub Packages package，并保留 tag、digest、版本和权限证据；
- 公开 Release 已经可见后，不重写公共历史，而通过 patch release 修复。

现有 `v1.5.0` release 仍是源码发布，因为公开 tag 早于 GHCR package 通道且不得移动。后续包含 release workflow 和 Dockerfile 支持的 release tag 会通过 GitHub Packages 发布 GHCR 容器镜像。安装包、签名二进制制品、provenance bundle 和包管理器发布仍属于后续版本范围。

## 发布通道

| 通道 | 归属 | 触发条件 | 输出 |
| --- | --- | --- | --- |
| CI | `.github/workflows/ci.yml` | pull request、推送到 `main`/`master`、手动触发 | Rust、CLI smoke、website、i18n 验证日志 |
| GitHub Release DI | `.github/workflows/release.yml` | 推送符合版本管理方案的 tag，或对已有 tag 手动触发 | GitHub Release、源码压缩包、`release-evidence-*` artifact |
| GitHub Packages DI | `.github/workflows/release.yml` | release tag 验证通过后，或对包含 package 支持的已有 tag 手动触发 | `ghcr.io/yetmos/eva-cli`、package digest、package metadata、包发布证据 |
| Website DI | `.github/workflows/pages.yml` | website、docs、assets、scripts 或 pages workflow 变更后推送 | GitHub Pages 部署 |

## 平台矩阵

每次发布都必须保持桌面平台矩阵为绿色。

| 平台 | GitHub runner | 必需 shell 假设 | 必需检查 |
| --- | --- | --- | --- |
| Windows | `windows-latest` | PowerShell Core、Windows 路径，release smoke 不依赖 POSIX-only 命令 | `cargo fmt`、`cargo clippy`、`cargo test`、CLI smoke、release gates |
| macOS | `macos-latest` | 仓库脚本使用 PowerShell Core，Cargo 侧遵循 POSIX 文件系统行为 | `cargo fmt`、`cargo clippy`、`cargo test`、CLI smoke、release gates |
| Linux | `ubuntu-latest` | 仓库脚本使用 PowerShell Core，作为网站和 Pages 构建的默认 CI 主机 | `cargo fmt`、`cargo clippy`、`cargo test`、CLI smoke、release gates、website build |

除非失败项已经明确写入非目标或实验项，否则任一平台失败都阻断发布。

## 分支与版本契约

- 发布 commit 默认进入 `main`，除非仓库默认分支发生变化。
- 根 `Cargo.toml` 的 package version 和 `[workspace.package].version` 必须与 release tag 去掉前缀 `v` 后一致。
- 正式 release tag 使用 `vMAJOR.MINOR.PATCH`，例如 `v1.5.0`。
- alpha 和 beta tag 遵循 [版本管理方案](版本管理方案.md)，例如 `v1.5.1-alpha` 或 `v1.5.1-beta.1`。
- 公开发布优先使用 annotated tag。
- GitHub Release 已经公开后，不重写公共 tag；使用 `v1.5.1` 之类的 patch release 修复。

## CI 门禁

CI 门禁在 pull request、主分支推送和手动触发时运行。它必须证明源码树可构建，并且公开命令面没有回退。

必需 CI 检查：

```powershell
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
./scripts/validate-version-management.ps1
./scripts/build-site-i18n.ps1
./scripts/validate-i18n.ps1
```

CI workflow 还会运行 CLI smoke 命令，覆盖 version、doctor、config validate、runtime inspect、basic task 执行、adapter/MCP/Skill 诊断、discovery、memory context、hardware 诊断、backup/snapshot、restore plan、upgrade check 和 release gates。

## DI 门禁

Release DI 门禁从不可变 release tag 运行。在全部平台验证 job 通过之前，不得发布 GitHub Release。

必需 release DI 检查：

```powershell
cargo run -- release check --output json
cargo run -- release security --output json
cargo run -- release perf --output json
cargo run -- release migration --output json
./scripts/validate-version-management.ps1 -Tag $env:RELEASE_TAG
```

publish job 会把每个命令输出采集到 `release-evidence/`，并上传为 `release-evidence-${RELEASE_TAG}`。该 artifact 是 tag 对应的机器可读发布证据。

对包含 package 支持的 release tag，`packages` job 必须在 release verify job 之后运行，并满足：

- workflow 权限包含 `contents: read` 和 `packages: write`；
- 默认使用 `GITHUB_TOKEN` 发布与本仓库关联的 package；
- 只有访问其他私有仓库 package 或跨仓库写入时，才允许使用最小权限 PAT；
- package 名称、版本、tag、digest 和 registry URL 必须写入 release evidence；
- 稳定 release 才能更新 `latest` tag；alpha/beta 只能发布 prerelease tag；
- 推送多平台 package 前，必须对构建出的容器镜像运行 `eva --version` smoke test；
- package 发布失败必须阻断该版本的 package 通道，但不能回写或移动已公开 release tag。

## 文档与网站门禁

当前项目以源码发布为主，公开命令契约高度依赖文档，因此文档变更属于发布相关变更。发布前必须：

- 更新发布说明、迁移指南、兼容性策略和对应版本的发版计划；
- 保持版本管理方案和当前人类可读版本标签与 `Cargo.toml` 对齐；
- 在 `docs/_i18n/manifest.json` 登记新增文档；
- 保持中文和英文文档路径符合现有 docs 树结构；
- 运行多语言网站构建和 i18n 校验脚本；
- 仅在 docs/site 构建成功后允许 Pages workflow 部署。

## 发布流程

1. 在 `main` 准备 release commit。
2. 本地运行 CI 和 DI 门禁章节列出的 preflight 检查。
3. 推送 release commit，并确认 Windows、macOS、Linux 三个平台 CI 全绿。
4. 创建 annotated release tag 并推送到 GitHub。
5. 确认 GitHub Release workflow 完成并上传 release evidence。
6. 对包含 package 支持的 tag，确认 `release-evidence/package-ghcr.json` 已记录 GHCR package digest。
7. 确认 GitHub Release 正文、源码压缩包、package 链接和文档链接都指向同一个 release tag。

## 二进制制品路线图

签名二进制打包不属于当前 V1.5 源码发布范围。GHCR 容器发布是已经实现的 GitHub Packages 通道，适用于包含 package 支持的 release tag；后续自动化发布可以从以下 target map 开始：

| 平台 | 初始 target | 后续输出 |
| --- | --- | --- |
| Windows | `x86_64-pc-windows-msvc` | GitHub Release `.zip`；具备签名后发布安装器 |
| macOS Intel | `x86_64-apple-darwin` | GitHub Release `.tar.gz`；具备签名/公证后发布 bundle |
| macOS Apple Silicon | `aarch64-apple-darwin` | GitHub Release `.tar.gz`；具备签名/公证后发布 bundle |
| Linux | `x86_64-unknown-linux-gnu` | GitHub Release `.tar.gz`；后续再接入 Linux 包管理器 |
| Container | `linux/amd64`、`linux/arm64` | GitHub Packages Container Registry：`ghcr.io/yetmos/eva-cli:<version>` |
| Ecosystem packages | npm、NuGet、Maven/Gradle、RubyGems 等 GitHub Packages 支持的 registry | 仅在对应 package metadata、安装验证和兼容策略完成后启用 |

在没有更新安全评审、兼容性策略和回滚流程前，不要把未签名安装器加入公开发布路径。

说明：GitHub Packages 不提供 Cargo crate registry 替代品。Rust crate 公共发布应另行评估 crates.io；GitHub Packages 用于容器镜像或 GitHub 已支持生态的 package registry。

## 失败与修复策略

如果 CI 在打 tag 前失败，修复 release 分支并重新运行 CI。不要从未通过平台矩阵的 commit 创建 release tag。

如果 release workflow 在 GitHub Release 可见前失败，先修复 commit，删除本地 tag；仅当没有公开 Release 时删除远端 tag，然后从修复后的 commit 重新创建 tag。

如果 GitHub Release 已经公开，保持已发布 tag 不可变，并发布 patch release。

## 完成证据

只有同时具备以下证据，才认为一次发布完成：

- release commit 上 Windows、macOS、Linux CI 全绿。
- release tag 上 Windows、macOS、Linux release workflow 全绿。
- GitHub Release 已为该 tag 创建。
- GitHub 自动生成的源码压缩包可下载。
- `release-evidence-${RELEASE_TAG}` 已上传。
- 对包含 package 支持的 tag：package registry 页面可访问，digest 与 release evidence 一致，拉取 smoke test 通过。
- 文档和网站校验成功完成。
