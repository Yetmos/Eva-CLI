# Eva-CLI 版本管理方案

日期：2026-07-06
范围：项目版本号、版本状态、GitHub 仓库分支/tag/release 管理

本文定义 Eva-CLI 的版本命名、递增规则和 GitHub 仓库版本管理流程。它补充
[项目发布方案](项目发布方案.md)，用于判断何时提升大版本、小版本、小小版本，以及如何在 GitHub 上管理分支、tag、release、milestone 和修复版本。

## 版本号格式

项目对外展示版本使用以下格式：

```text
V<major>.<minor>.<patch>-<status>
```

示例：

```text
V1.0.15-alpha
```

字段含义如下：

| 字段 | 示例值 | 中文名称 | 递增条件 |
| --- | --- | --- | --- |
| `major` | `1` | 大版本号 | 发生大规模重构、架构边界重划、核心运行时更新、公开契约不兼容变更，或多个模块同时进入新的稳定阶段时递增。 |
| `minor` | `0` | 小版本号 | 有小范围功能新增、非核心功能删除、功能组合调整、命令面扩展，且整体架构和核心契约仍保持兼容时递增。 |
| `patch` | `15` | 小小版本号 | 对已有功能做迭代、修复、体验改进、文档补齐、诊断增强、兼容性补丁或内部实现优化时递增。 |
| `status` | `alpha` | 版本状态 | 表示当前版本处于内部验证、外部验证或正式发布阶段。允许值为 `alpha`、`beta`、`release`。 |

递增某个高位版本号时，低位版本号归零。例如 `V1.4.9-release` 进入新的小版本后为 `V1.5.0-alpha`；`V1.9.12-release` 进入新的大版本后为 `V2.0.0-alpha`。

## 版本状态

| 状态 | 含义 | GitHub Release 设置 | 适用范围 |
| --- | --- | --- | --- |
| `alpha` | 内部验证版本，功能可能不完整，允许调整命令、配置、文档结构和实现边界。 | 勾选 prerelease，不设为 latest。 | 设计验证、开发者试用、CI/DI 流程验证。 |
| `beta` | 外部验证版本，功能基本冻结，只接受阻断问题、兼容性问题、文档缺口和发布门禁修复。 | 勾选 prerelease，不设为 latest。 | 公开候选验证、迁移演练、平台矩阵检查。 |
| `release` | 正式发布版本，必须通过发布门禁并具备可审计发布证据。 | 不勾选 prerelease，可设为 latest。 | 面向用户的稳定版本。 |

`alpha` 可以继续新增或删除功能；`beta` 原则上不再引入新功能；`release` 只允许通过后续 patch 版本修复，不修改已公开 tag。

## Git、Cargo 与展示版本映射

不同系统对版本格式的要求不同，必须按以下规则映射：

| 场景 | 格式 | 示例 |
| --- | --- | --- |
| 文档、路线图、milestone、release 标题 | `V<major>.<minor>.<patch>-<status>` | `V1.0.15-alpha` |
| Git tag，alpha/beta | `v<major>.<minor>.<patch>-<status>` 或 `v<major>.<minor>.<patch>-<status>.<n>` | `v1.0.15-alpha`、`v1.0.15-beta.1` |
| Git tag，正式 release | `v<major>.<minor>.<patch>` | `v1.0.15` |
| Cargo package version，alpha/beta | `<major>.<minor>.<patch>-<status>` 或 `<major>.<minor>.<patch>-<status>.<n>` | `1.0.15-alpha` |
| Cargo package version，正式 release | `<major>.<minor>.<patch>` | `1.0.15` |

说明：

- 人类可读版本可以显式写成 `V1.0.15-release`，表示状态为正式发布。
- Git tag 和 Cargo 稳定版本不使用 `-release` 后缀，避免把正式版本误标为预发布版本。
- Git tag 使用小写 `v` 前缀，延续现有发布方案；文档标题和路线图可使用大写 `V`。
- 同一 tag 不得重复发布。需要再次发布同一阶段时，使用 `alpha.2`、`beta.2`，或递增 patch。

## 版本递增决策

每个合并到发布分支的 PR 都必须判断版本影响级别。

### 提升大版本号

满足任一条件时提升 `major`：

- 发生核心架构重构，影响运行时边界、模块依赖、配置契约或数据持久化格式；
- 删除、重命名或不兼容修改已经正式发布的 CLI 命令、配置字段、输出 JSON schema；
- 更换关键运行模型，例如任务调度、Agent 生命周期、权限模型、Adapter/MCP/Skill 接入边界；
- 迁移成本需要用户执行明确步骤，且无法通过兼容层自动处理。

示例：`V1.8.4-release -> V2.0.0-alpha`。

### 提升小版本号

满足任一条件时提升 `minor`：

- 新增较完整的功能面，例如新增一组 CLI 子命令、一个能力域或一个发布门禁；
- 删除仍处于 `alpha`/`beta` 的实验能力，且未破坏正式发布契约；
- 调整功能组合或默认策略，但保留兼容入口和迁移说明；
- 完成一个路线图阶段，且用户可感知能力有明显扩展。

示例：`V1.0.15-release -> V1.1.0-alpha`。

### 提升小小版本号

满足任一条件时提升 `patch`：

- 对已有功能做迭代优化、错误修复、诊断补充、日志改进或文档补齐；
- 修复平台兼容性、CI/DI 门禁、发布说明、迁移指南或网站构建问题；
- 内部实现优化不改变公开命令、配置、schema 和用户工作流；
- 对 `alpha`/`beta` 阶段同一版本进行验证修正。

示例：`V1.0.15-alpha -> V1.0.16-alpha` 或 `V1.0.15-beta.1 -> V1.0.15-beta.2`。

## GitHub 仓库版本管理方案

### 分支模型

| 分支 | 用途 | 规则 |
| --- | --- | --- |
| `main` | 默认集成分支和正式发布来源 | 必须保持 CI 绿色；所有发布 tag 默认从 `main` 或已合并到 `main` 的发布 commit 创建。 |
| `feature/<topic>` | 功能开发 | 通过 PR 合并；PR 描述必须声明版本影响级别。 |
| `fix/<topic>` | 缺陷修复 | 通过 PR 合并；默认提升 `patch`，除非改变公开契约。 |
| `docs/<topic>` | 文档更新 | 通过 PR 合并；发布相关文档必须同步 manifest 和索引。 |
| `release/v<major>.<minor>` | 发布准备分支，可选 | 只在需要冻结某个小版本时创建；只接受门禁修复、文档修复和版本号修正。 |
| `hotfix/v<major>.<minor>.<patch>` | 已发布版本紧急修复，可选 | 从对应 release tag 分出，修复后合并回 `main`，并发布新的 patch tag。 |

默认流程保持轻量：日常开发走 `main` + PR；只有进入 beta/release 冻结或需要维护已发布版本时才创建 `release/*` 或 `hotfix/*`。

### Tag 规则

- alpha/beta tag：`v1.0.15-alpha`、`v1.0.15-beta.1`。
- 正式 release tag：`v1.0.15`。
- 公开 tag 必须使用 annotated tag，并写明版本状态、核心变更和验证摘要。
- tag 一经公开，不得 force push、删除后重建或指向新 commit。
- tag 对应的 `Cargo.toml`、`Cargo.lock`、发布说明和 release evidence 必须属于同一个 commit。

### GitHub Release 规则

每个 GitHub Release 必须绑定一个不可变 tag，并按状态设置：

- `alpha`/`beta`：勾选 prerelease，不设为 latest。
- `release`：不勾选 prerelease；只有最新稳定版设置为 latest。
- Release 标题使用人类可读版本，例如 `Eva-CLI V1.0.15-alpha` 或 `Eva-CLI V1.0.15-release`。
- Release 正文必须包含变更摘要、兼容性说明、迁移说明、验证证据、已知问题和文档链接。
- GitHub 自动生成的源码压缩包可作为源码发布制品；二进制制品只有在签名、校验和、平台矩阵与回滚方案完成后才加入。

### Milestone、Issue 和 PR 管理

- GitHub milestone 使用人类可读版本名，例如 `V1.0.15-alpha`、`V1.0.15-beta`、`V1.0.15-release`。
- Issue 使用标签区分影响级别：`version:major`、`version:minor`、`version:patch`。
- Issue 使用标签区分版本状态：`status:alpha`、`status:beta`、`status:release-blocker`。
- PR 必须在描述中声明：
  - 版本影响级别：major/minor/patch/no-version-impact；
  - 是否影响 CLI 命令、配置、JSON 输出、文档、CI/DI、网站；
  - 已运行的验证命令；
  - 是否需要迁移说明或发布说明。
- 合并前如果 PR 被标记为 `version:major` 或 `status:release-blocker`，必须补充发布说明、迁移指南或兼容性策略。

### 版本发布流程

1. 确认目标版本，例如 `V1.0.15-alpha`。
2. 检查 milestone 中所有 release blocker 已关闭或转移。
3. 更新版本号、发布说明、迁移说明、兼容性说明和相关文档。
4. 运行 CI 门禁、release 门禁、文档构建和 i18n 校验。
5. 合并 release commit 到 `main`，确认平台矩阵全绿。
6. 创建 annotated tag：

```powershell
git tag -a v1.0.15-alpha -m "Eva-CLI V1.0.15-alpha"
git push origin v1.0.15-alpha
```

正式版本使用：

```powershell
git tag -a v1.0.15 -m "Eva-CLI V1.0.15-release"
git push origin v1.0.15
```

7. 等待 GitHub Release workflow 完成。
8. 检查 GitHub Release 的 prerelease/latest 状态、正文、源码包和 release evidence。
9. 关闭对应 milestone，并创建下一阶段 milestone。

### 修复与回滚

- 如果 tag 创建前发现问题，修复 release commit 后再创建 tag。
- 如果 alpha/beta release 已公开，优先发布新的 `alpha.N`、`beta.N` 或 patch 版本，不复写原 tag。
- 如果正式 release 已公开，禁止修改原 tag；必须发布新的 patch release，例如 `v1.0.16`。
- 如果发布流程失败但 GitHub Release 尚未公开，可删除本地 tag；远端 tag 仅在确认没有公开 Release 和下游引用时删除。
- 所有回滚都必须记录原因、影响范围和替代版本，不能只依赖 GitHub UI 操作。
