# 网站与文档多语言适配方案

## 目标

Eva-CLI 官网和文档需要从当前中文内容为主，迁移为英语主语言、多语言可扩展的发布结构。最终状态应满足：

- 目标状态下，英语详案成为 canonical 内容源，默认站点入口 `/` 和默认文档入口 `/docs/` 面向英语读者。
- 多语言能力按 BCP 47 locale code 扩展，覆盖全球主要语言和书写方向；中文简体只是第一批翻译语言之一。
- 官网保持零运行时依赖，发布产物仍是静态 HTML、Markdown 和资源文件；当语言数超过 3 种时，使用构建期静态生成，避免手写几十份重复 HTML。
- 每个页面都有明确的 `lang`、`canonical`、`alternate hreflang` 和同页语言切换入口。
- 文档迁移期间保留旧链接的可读性，避免一次性改名导致现有链接全部失效。

## 当前覆盖规则

当前迁移尚未完成：英文文档已经承担默认公开入口和稳定 slug 职责，但多数英文页面仍是摘要；
`docs/zh-CN/` 保留完整原始详案，是当前架构细节和实现规格的事实主源。

在英文详案补齐前：

- 架构细节、API 语义、实现规格的结构性更新先进入 `docs/zh-CN/`。
- 英文文档同步为摘要或全文时，必须在 `docs/_i18n/manifest.json` 中诚实标记覆盖度。
- 不能用英文摘要覆盖中文详案中的具体约束。
- `defaultLocale: "en"` 只表示站点默认入口，不表示英文内容已经是完整事实主源。

## 现状约束

当前发布流程由 `.github/workflows/pages.yml` 完成：

1. 将 `website/` 复制到 `public/`。
2. 将 `docs/` 复制到 `public/docs/`。
3. 将 `website/docs/` 再复制到 `public/docs/`，用于覆盖或补充文档首页。
4. 将 `assets/` 复制到 `public/assets/`。

当前结构是：

```text
website/
  index.html
  styles.css
  docs/index.html
docs/
  README.md
  *.md
assets/
  eva-cli-logo.svg
  eva-cli-architecture.svg
  eva-cli-architecture.zh-CN.svg
```

因此多语言方案必须避免 `docs/` 与 `website/docs/` 在同一路径下互相覆盖。`website/docs/` 应只承担文档站入口页面，真正的 Markdown 文档内容放在 `docs/<locale>/` 中。

## 目标目录结构

推荐迁移后的结构如下：

```text
website/
  _templates/
    home.html                   # 官网首页模板
    docs-index.html             # 文档入口模板
  _i18n/

    ar.json                     # 其他语言按 locale 扩展
  index.html                    # 构建产物：英文官网首页，对应 /

  styles.css                    # 多语言共享样式

  README.md                     # 文档维护说明与语言入口，不再承载正文目录
```

英文文档文件名统一使用 kebab-case 英文 slug。非英文文档可以保留本地语言文件名，也可以复用英文 slug；无论采用哪种方式，都必须通过 `_i18n/manifest.json` 维护文档 ID、locale、翻译状态和路径映射。后续如果需要改 URL，应保留旧路径入口一个发布周期。

## URL 策略

| 页面类型 | 英文默认 URL | 本地化页面 |
| --- | --- | --- |
| 官网首页 | `/` | `/<locale>/` |
| 文档入口 | `/docs/` | `/docs/<locale>/README.md` |
| 架构总览 | `/docs/en/architecture/architecture-overview.md` | `/docs/<locale>/<mapped-path>.md` |
| 其他文档 | `/docs/en/<slug>.md` | `/docs/<locale>/<mapped-path>.md` |

不建议使用 query 参数承载语言，例如 `?lang=zh-CN`。路径式语言更适合静态站点、搜索引擎索引、分享链接和 GitHub Pages 缓存。

默认语言策略：

- `/` 必须是英语，因为英语是主语言。
- `/docs/` 必须是英语文档入口，同时提供所有可用语言入口。
- 浏览器语言自动跳转只作为增强能力，不应替代明确 URL。首次访问可以显示语言入口或轻量提示，但不要强制跳转。

## 全球语言覆盖策略

“支持全球大多数语言”不应理解为首日人工维护所有语言的完整翻译，而应分成能力覆盖和内容覆盖两层：

- 能力覆盖：站点、文档目录、语言切换、排版、SEO、CI 和发布流程必须支持任意 BCP 47 locale。
- 内容覆盖：按优先级逐步补齐翻译，允许某些语言只有官网首页、快速开始或核心架构文档。

### Locale 规范

所有语言代码使用 BCP 47：

- 语言级 locale：`en`、`fr`、`de`、`ja`、`ko`、`ar`。
- 地区变体：`zh-CN`、`zh-TW`、`pt-BR`、`es-419`。
- 脚本变体：必要时使用 `sr-Cyrl`、`sr-Latn` 这类脚本标签。

目录名、`html lang`、`hreflang` 和 `manifest.json` 中的 locale code 必须一致。不要混用 `cn`、`zh_cn`、`zh-Hans` 和 `zh-CN`。

### 推荐覆盖批次

第一批建议覆盖全球开发者常用语言和当前中文内容：

| 批次 | Locale | 说明 |
| --- | --- | --- |
| P0 | `en` | 默认入口、目标英文详案源 |
| P1 | `zh-CN`、`zh-TW`、`ja`、`ko` | 东亚主要开发者社区 |
| P1 | `es`、`pt-BR`、`fr`、`de` | 欧洲与拉美主要覆盖 |
| P1 | `ru`、`ar`、`hi` | 欧亚、中东、南亚基础覆盖 |
| P2 | `id`、`vi`、`th`、`tr`、`it`、`pl`、`nl` | 增量开发者社区和区域增长市场 |
| P2 | `uk`、`fa`、`he`、`ms`、`bn`、`ur` | 按贡献者、下载量或访问数据扩展 |
| P3 | 其他 locale | 社区维护或按产品数据追加 |

P0/P1 应覆盖官网首页、文档入口、README、总体架构和快速开始类内容。P2/P3 可以先覆盖官网首页和文档入口，再逐步补核心文档。

### 书写方向

`manifest.json` 必须为每个 locale 声明 `dir`：

- `ltr`：英语、中文、日语、韩语、西班牙语、法语等。
- `rtl`：阿拉伯语、希伯来语、波斯语、乌尔都语等。

HTML 必须同时设置：

```html
<html lang="ar" dir="rtl">
```

CSS 必须优先使用逻辑属性，减少左右方向硬编码：

```css
.content {
  margin-inline: auto;
  padding-inline: 16px;
  border-inline-start: 4px solid var(--accent);
}
```

避免在文案中硬编码箭头方向，例如用 `Next`/`Previous` 的本地化文案或图标镜像策略，而不是固定 `→` 表示前进。

### 本地化格式

多语言文案不仅是文本翻译，还包括格式本地化：

- 日期：使用 locale 格式，例如 `Intl.DateTimeFormat(locale)`。
- 数字：使用 `Intl.NumberFormat(locale)`。
- 复数：需要资源文件支持 plural key，而不是简单拼接 `1 item` / `n items`。
- 标点：中文、日文、阿拉伯语等应使用符合本地阅读习惯的标点。
- 术语：保留 `Runtime`、`EventBus`、`Adapter`、`MCP` 等专有技术词时，应在各语言 glossary 中统一规则。

官网静态生成时可以在构建期完成这些格式；如果没有构建期脚本，则避免在静态 HTML 中出现需要动态复数和日期的文案。

## 内容源与翻译关系

目标状态下，英文详案应成为统一内容源。当前迁移窗口内，以 `docs/_i18n/manifest.json` 的 `contentAuthority.locale` 指向的 locale 作为事实主源，避免不同语言版本各自演进。

建议新增 `docs/_i18n/manifest.json`：

```json
{
  "defaultLocale": "en",
  "locales": [
    {
      "code": "en",
      "label": "English",
      "nativeLabel": "English",
      "dir": "ltr",
      "tier": "P0",
      "isDefault": true
    },
    {
      "code": "zh-CN",
      "label": "简体中文",
      "nativeLabel": "简体中文",
      "dir": "ltr",
      "tier": "P1",
      "isDefault": false
    },
    {
      "code": "ja",
      "label": "Japanese",
      "nativeLabel": "日本語",
      "dir": "ltr",
      "tier": "P1",
      "isDefault": false
    },
    {
      "code": "ar",
      "label": "Arabic",
      "nativeLabel": "العربية",
      "dir": "rtl",
      "tier": "P1",
      "isDefault": false
    }
  ],
  "documents": [
    {
      "id": "architecture-overview",
      "source": "docs/en/architecture/architecture-overview.md",
      "translations": {
        "zh-CN": "docs/zh-CN/architecture/总体架构方案.md",
        "ja": "docs/ja/architecture-overview.md",
        "ar": "docs/ar/architecture-overview.md"
      },
      "status": {
        "zh-CN": "current",
        "ja": "needs-review",
        "ar": "missing"
      }
    }
  ],
  "fallbacks": {
    "zh-HK": "zh-TW",
    "zh-SG": "zh-CN",
    "pt-PT": "pt-BR",
    "es-MX": "es"
  }
}
```

翻译状态建议使用以下枚举：

- `current`：与英文源同步。
- `needs-review`：已翻译，但需要人工审阅。
- `stale`：英文源已更新，翻译未同步。
- `missing`：目标语言暂缺。
- `partial`：只翻译了摘要、导航或关键章节。

文档顶部建议增加轻量元信息区，方便读者识别语言和翻译状态：

```markdown
> Language: English
> Canonical: docs/en/architecture/architecture-overview.md
> Translations: [简体中文](../architecture/总体架构方案.md), 日本語: docs/ja/architecture/architecture-overview.md
```

翻译文档顶部对应：

```markdown
> Language: 简体中文
> English default entry: ../../en/architecture/architecture-overview.md
> Translation status: current
```

### 翻译回退策略

页面缺失时按以下顺序回退：

1. 精确 locale，例如 `pt-BR`。
2. `manifest.json` 中声明的地区回退，例如 `pt-PT -> pt-BR`。
3. 语言级回退，例如 `es-419 -> es`。
4. 英文默认入口。

回退不能静默发生。页面或语言切换菜单必须标记当前内容来自哪个语言，例如：

```text
This page is not available in Arabic yet. Showing English instead.
```

### 术语表

建议新增 `docs/_i18n/glossary.json`，统一技术术语：

```json
{
  "terms": [
    {
      "id": "eventbus",
      "source": "EventBus",
      "translations": {
        "zh-CN": "EventBus",
        "ja": "EventBus",
        "ar": "EventBus"
      },
      "note": "Keep as product architecture term."
    },
    {
      "id": "adapter",
      "source": "Adapter",
      "translations": {
        "zh-CN": "Adapter",
        "ja": "Adapter",
        "ar": "Adapter"
      }
    }
  ]
}
```

术语表优先约束架构名词、配置字段、API 名称、manifest 字段、错误码和 CLI 命令。代码、命令、Topic、JSON key、Lua binding 名称不得翻译。

### 本地化图片资源

包含可读文字的内容型图片必须和文档一样纳入多语言管理。英文图片作为默认图片源，其他语言图片按 locale code 维护独立文件，并登记到 `docs/_i18n/manifest.json` 的 `assets` 区域：

```json
{
  "id": "architecture-diagram",
  "source": "assets/eva-cli-architecture.svg",
  "translations": {
    "zh-CN": "assets/eva-cli-architecture.zh-CN.svg"
  },
  "status": {
    "zh-CN": "current"
  }
}
```

官网模板在构建期读取该映射，为每个语言首页生成对应的 `<img>` 和背景图 URL。文档 Markdown 直接引用本语言图片，例如中文文档引用 `../../assets/eva-cli-architecture.zh-CN.svg`，英文文档引用 `../../assets/eva-cli-architecture.svg`。Logo 等品牌中性资源不需要本地化。

## 官网适配方案

### 页面拆分

官网页面数量少时可以手写 HTML；但如果目标是覆盖全球大多数语言，应采用“模板 + locale JSON + 构建期静态生成”的方式。发布产物仍然是纯静态 HTML，不需要运行时 i18n 框架。

- `website/_templates/home.html`：官网首页模板。
- `website/_templates/docs-index.html`：文档入口模板。
- `website/_i18n/<locale>.json`：每种语言的 UI 文案、按钮、meta、导航和 CTA。
- `website/index.html`：生成后的英文首页。
- `website/<locale>/index.html`：生成后的本地化首页。
- `website/docs/index.html`：生成后的英文文档入口，链接到英文默认文档，并提供所有语言入口。

这样做的好处是：

- GitHub Pages 可直接发布，不需要 Node、Vite、Next.js 或额外构建链。
- 每个页面可以独立设置 SEO 元信息。
- 全语言共享模板，避免 20 种以上语言时重复维护 HTML 结构。
- 可以集中校验缺失 key、错误 `hreflang`、RTL 方向和链接映射。

构建脚本可以从最小实现开始：

- PowerShell：`scripts/build-site-i18n.ps1`，适合当前 Windows 开发环境和 GitHub Actions。
- 或 Node：`scripts/build-site-i18n.mjs`，适合后续更复杂的模板替换和 ICU plural。

在未引入成熟站点生成器前，模板语法保持简单，例如 `{{title}}`、`{{nav.docs}}`、`{{dir}}`。不要在模板里写复杂逻辑，复杂逻辑放在构建脚本和 manifest 中。

### HTML 元信息

每个 HTML 页面都必须设置：

```html
<html lang="en">
<link rel="canonical" href="https://Eva-CLI.com/">
<link rel="alternate" hreflang="en" href="https://Eva-CLI.com/">
<link rel="alternate" hreflang="zh-CN" href="https://Eva-CLI.com/zh-CN/">
<link rel="alternate" hreflang="x-default" href="https://Eva-CLI.com/">
```

本地化页面对应：

```html
<html lang="ar" dir="rtl">
<link rel="canonical" href="https://Eva-CLI.com/ar/">
<link rel="alternate" hreflang="en" href="https://Eva-CLI.com/">
<link rel="alternate" hreflang="zh-CN" href="https://Eva-CLI.com/zh-CN/">
<link rel="alternate" hreflang="ar" href="https://Eva-CLI.com/ar/">
<link rel="alternate" hreflang="x-default" href="https://Eva-CLI.com/">
```

同一组页面的 `title`、`description`、Open Graph 和 Twitter Card 内容也要随语言本地化。不要只翻译正文而保留其他语言的 meta。

当支持几十种语言时，每个页面的 `alternate hreflang` 必须由 manifest 自动生成，不能手写。P2/P3 语言如果某篇文档缺失，不应生成对应 `hreflang`，否则搜索引擎会把回退页误判为该语言内容。

### 语言切换

站点顶部导航增加语言切换入口。语言列表较短时可直接显示：

```text
English | 简体中文
```

语言列表超过 6 种时改为语言菜单：

```text
Language: English
  English
  简体中文
  繁體中文
  日本語
  한국어
  Español
  Português do Brasil
  Français
  Deutsch
  Русский
  العربية
```

语言切换必须跳到同等语义页面，而不是只跳到语言首页。例如：

- `/docs/en/architecture/architecture-overview.md` 切换到 `/docs/zh-CN/architecture/总体架构方案.md`。
- `/zh-CN/` 切换到 `/`。

如果目标语言页面缺失，才退回目标语言入口页，并在页面上标记“该翻译暂未提供”。

### 样式与排版

共享 `styles.css`，但需要为不同语言留出文本长度弹性：

- 按内容自然换行，不为中文或英文写死宽度。
- 按钮和导航使用 `min-height`、`max-width`、`overflow-wrap: anywhere` 防止英文长词溢出。
- 避免依赖中文短句长度设计卡片高度；英文通常更长。
- 字体栈保持系统字体优先，避免新增远程字体依赖。
- 使用逻辑属性支持 RTL：`margin-inline`、`padding-inline`、`inset-inline-start`、`border-inline-start`。
- 对 CJK、拉丁、阿拉伯、天城文、泰文分别检查导航、按钮、卡片和表格换行。
- 不在 CSS 中按语言写大量选择器；确实需要语言微调时，只允许少量 `[lang|="ja"]`、`[dir="rtl"]` 级别规则。

| `hardware-hotplug` | `hardware-hotplug.md` | `外接硬件接入与热插拔架构方案.md` |
| `project-configuration` | `project-configuration.md` | `项目配置方案.md` |
| `process-level-upgrade` | `process-level-upgrade.md` | `进程级停机升级架构方案.md` |
| `design-risk-review` | `design-risk-review.md` | `方案设计风险评审.md` |

### 中文文档迁移

中文文档第一阶段可以从 `docs/*.md` 原地复制到 `docs/zh-CN/`，再逐步在旧位置留下入口说明。推荐分两步：

1. 首个多语言版本：复制中文文档到 `docs/zh-CN/`，旧路径暂时保留，避免链接失效。
2. 下个稳定版本：旧路径改为短说明，指向英文默认入口和中文新路径。

旧路径说明示例：

```markdown
# 文档已迁移

English default entry: [Architecture Overview](../../en/architecture/architecture-overview.md)
简体中文： [总体架构方案](../architecture/总体架构方案.md)
```

### 其他语言文档迁移

其他语言不建议一次性复制全部文档目录后留空。推荐按覆盖度逐步创建：

1. 每个 locale 先创建 `docs/<locale>/README.md`，说明当前覆盖状态。
2. P1 语言优先翻译官网首页、文档入口、`README.md`、`architecture-overview.md` 和快速开始类文档。
3. 每新增一篇翻译，都在 `docs/_i18n/manifest.json` 中登记路径和状态。
4. 翻译未完成时不要创建空文件；用回退策略指向英文默认入口。

每种语言 README 顶部建议标记覆盖度：

```markdown
# Eva-CLI Documentation

Language: Deutsch
Coverage: partial
English default entry: ../en/README.md
Missing pages fall back to English.
```

### README 分工

`docs/README.md` 不再作为某一种语言的正文目录，而是作为文档维护入口：

- 说明默认语言为英语。
- 链接 `docs/en/README.md` 和所有已启用 locale 的 `README.md`。
- 说明翻译状态、贡献规则和文档 ID 规范。

`docs/<locale>/README.md` 分别维护对应语言的阅读顺序。未翻译文档必须显示英文默认入口链接，而不是隐藏。

## 发布流程调整

当前 GitHub Pages 工作流可以继续使用，但应加一个校验步骤，避免语言文件缺失或被覆盖：

```yaml
- name: Validate i18n structure
  run: |
    test -f website/index.html
    test -f website/docs/index.html
    test -f docs/_i18n/manifest.json
    test -d docs/en
    test -f website/_i18n/en.json
```

后续可增加一个无依赖脚本，例如 `scripts/validate-i18n.ps1` 或 `scripts/validate-i18n.mjs`，校验：

- `manifest.json` 中声明的文件都存在。
- 每篇英文默认文档至少声明所有启用 locale 的翻译状态。
- HTML 页面包含正确的 `lang`、`canonical` 和 `hreflang`。
- 网站导航中的语言切换链接可访问。
- 不存在 `website/docs/` 覆盖 `docs/en/` 或 `docs/zh-CN/` 的同名路径。
- `dir=rtl` 的 locale 生成 RTL 页面，且 CSS 不依赖无法镜像的左右硬编码。
- `website/_i18n/<locale>.json` 不缺少模板必需 key。
- 未翻译页面不会生成错误的 `hreflang`。

Pages 构建步骤建议调整为：

```yaml
- name: Build localized website
  run: ./scripts/build-site-i18n.ps1

- name: Validate i18n structure
  run: ./scripts/validate-i18n.ps1

- name: Build static site
  run: |
    mkdir -p public/docs
    cp -R website/. public/
    cp -R docs/. public/docs/
    cp -R website/docs/. public/docs/
    cp -R assets public/assets
```

如果生成产物会写入 `website/index.html`、`website/<locale>/index.html` 和 `website/docs/index.html`，需要确保模板目录 `_templates/` 和资源目录 `_i18n/` 不被发布到公开站点；可以在复制后删除：

```sh
rm -rf public/_templates public/_i18n
```

## 迁移步骤

### 第一阶段：建立结构

1. 新增 `docs/en/`、`docs/zh-CN/`、`docs/_i18n/manifest.json` 和 `docs/_i18n/glossary.json`。
2. 将当前中文 Markdown 复制到 `docs/zh-CN/`。
3. 为每篇文档确定英文 slug 和文档 ID。
4. 更新 `docs/README.md` 为语言入口和维护说明。
5. 在 manifest 中登记 P0/P1 locale，P2/P3 先只登记计划，不生成页面。

验收标准：

- `/docs/zh-CN/中文文档入口.md` 能看到当前中文文档目录。
- `manifest.json` 能完整覆盖现有文档。
- 旧 `docs/*.md` 链接暂不失效。
- 每个启用 locale 都有 `code`、`nativeLabel`、`dir`、`tier` 和翻译状态。

### 第二阶段：英文默认入口与目标详案

1. 翻译并重写 `docs/en/*.md`，让英文版本成为 authoritative source。
2. 新增 `website/_templates/` 和 `website/_i18n/en.json`。
3. 生成 `website/index.html` 和 `website/docs/index.html`。
4. 为官网页面补齐 `canonical`、`hreflang`、本地化 meta 和语言切换。

验收标准：

- `/` 是完整英文首页。
- `/docs/` 指向英文文档入口。
- 英文页面不残留中文正文、中文按钮或中文 meta。
- 英文目标详案文档覆盖所有现有中文方案。

### 第三阶段：启用 P1 语言

1. 为 `zh-CN`、`zh-TW`、`ja`、`ko`、`es`、`pt-BR`、`fr`、`de`、`ru`、`ar`、`hi` 创建 `website/_i18n/<locale>.json`。
2. 为 P1 语言生成 `website/<locale>/index.html`。
3. 为 P1 语言创建 `docs/<locale>/README.md`。
4. 优先翻译架构总览和 README；未覆盖文档显示英文回退。
5. 对 `ar` 等 RTL 语言做布局检查。

验收标准：

- P1 官网首页都能访问。
- P1 文档入口都能看到覆盖状态和英文回退。
- `hreflang` 只为真实存在的本地化页面生成。
- RTL 页面设置 `dir="rtl"` 且核心布局不破。

### 第四阶段：链接与搜索优化

1. 更新 `README.md` 中的官网和文档入口，优先指向英文默认入口。
2. 在所有翻译文档顶部增加英文默认入口链接。
3. 在英文文档顶部增加可用翻译链接。
4. 增加 `sitemap.xml` 和 `robots.txt`，按语言列出 URL。
5. 让 Firebase Analytics 或后续分析工具按路径区分语言访问。

验收标准：

- 搜索引擎看到英文为默认入口。
- 读者可以从任意语言页面切换到对应语义页面或明确回退页。
- 旧文档链接有清晰迁移提示。

### 第五阶段：自动化校验与持续扩展

1. 在 CI 中加入 i18n 结构校验。
2. 增加链接校验，至少校验仓库内相对链接。
3. 增加翻译状态校验，英文源更新时把对应翻译标记为 `stale`。
4. 把 i18n 校验列为 Pages 发布前置步骤。
5. 根据访问数据、社区贡献和下载来源扩展 P2/P3 locale。

验收标准：

- 缺少翻译文件、错误路径、错误 `lang`、错误 `dir` 或覆盖冲突会导致 CI 失败。
- Pages 发布产物中 `/docs/en/`、P1 `/docs/<locale>/` 和 P1 `/<locale>/` 都存在。
- 新增 locale 不需要复制 HTML 结构，只需要 manifest、locale JSON 和对应翻译文件。

## 维护规则

- 新文档必须先按当前 `contentAuthority.locale` 创建事实主源；英文默认入口同步为摘要或全文。
- 文档 ID 一旦发布，不随标题调整而变化。
- 英文 slug 一旦发布，除非有迁移入口，否则不改名。
- 各语言翻译允许更贴近本地读者，但不得改变架构结论、约束和 API 语义。
- 英文源发生结构性修改时，相关翻译状态必须从 `current` 改为 `stale` 或 `needs-review`。
- 官网新增页面时必须同时登记语言切换目标；暂缺翻译时显示目标语言入口，不生成空页面。
- 新增 locale 必须先进入 manifest，并声明 `tier`、`dir`、`nativeLabel`、fallback 和覆盖范围。
- P0/P1 语言缺少官网首页或文档入口时，CI 应失败；P2/P3 可允许 `partial`，但必须有明确回退。
- 代码、命令、配置字段、JSON key、Topic、Lua binding、文件路径和错误码不得翻译。

## 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 旧中文路径直接移动导致外部链接失效 | 已分享链接不可访问 | 第一阶段保留旧路径，第二阶段再改为迁移入口 |
| `website/docs/` 覆盖 `docs/` 内容 | 发布产物缺文件或入口错乱 | 约束 `website/docs/` 只放入口 HTML，不放同名文档目录 |
| 各语言内容各自演进 | 架构结论不一致 | 用 `manifest.json`、`contentAuthority` 和覆盖状态绑定事实源文档 |
| 英文文本更长导致页面布局溢出 | 移动端或导航错位 | 共享样式增加换行、弹性宽度和按钮自适应 |
| RTL 语言布局错位 | 阿拉伯语、希伯来语等不可用 | 使用 `dir`、CSS 逻辑属性和 RTL 截图检查 |
| 语言数量过多导致手写页面不可维护 | HTML 结构分叉、SEO 标签遗漏 | 使用模板 + locale JSON + 构建期静态生成 |
| 未翻译页面被标记为本地语言 | 搜索引擎和读者误判内容语言 | 缺失翻译不生成 `hreflang`，页面显式显示英文回退 |
| 术语翻译不一致 | API、架构概念和配置说明难以理解 | 用 `glossary.json` 统一技术术语和不可翻译项 |
| 自动语言跳转影响 SEO | 搜索引擎索引混乱 | 默认路径固定为英语，只提供显式语言切换 |

## 推荐落地顺序

优先顺序如下：

1. 先建立 `docs/en/`、`docs/zh-CN/`、`docs/_i18n/manifest.json` 和 `docs/_i18n/glossary.json`。
2. 再把官网改成模板 + locale JSON + 静态生成，先生成英文和中文简体。
3. 然后补齐全部英文目标详案文档。
4. 再启用 P1 语言：首页、文档入口、README、架构总览优先。
5. 最后加 CI 校验、sitemap、RTL 检查和 P2/P3 扩展规则。

这样可以在不破坏当前 GitHub Pages 发布方式的前提下，逐步把 Eva-CLI 迁移到英语主语言、全球多语言可维护的结构。真正上线几十种语言时，不应继续手写 HTML 副本；应依赖 manifest、模板和自动校验保证规模化维护。
