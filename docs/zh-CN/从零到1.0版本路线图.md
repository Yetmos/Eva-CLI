# 从零到 1.0 版本路线图

Eva-CLI 从架构设计走向 1.0 发布，应按阶段推进，而不是直接从文档跳到大规模
Runtime 实现。每个阶段都要留下可评审、可测试、可版本化的交付物，并成为下一阶段
的基础。

截至 2026-07-01，仓库已经从纯文档阶段推进到 Rust workspace 骨架阶段：

- 中文架构详案、英文公开入口、官网与 wiki 结构已经基本齐备。
- 根目录 `Cargo.toml` 已建立 workspace，`crates/` 下已有 `eva-core`、`eva-config`、
  `eva-policy`、`eva-eventbus`、`eva-scheduler`、`eva-agent`、`eva-lua-host`、
  `eva-capability`、`eva-adapter`、`eva-mcp`、`eva-discovery`、`eva-memory`、
  `eva-hardware`、`eva-backup`、`eva-lifecycle`、`eva-runtime`、`eva-cli` 等边界 crate。
- `eva-core` 已落地 Topic、Event、Invoke、Capability、ID 和结构化错误等基础契约，
  并带有聚焦单元测试。
- `config/schemas/` 已放置 Agent、Adapter、Capability、Eva、Policy、Routes 等
  JSON Schema 文件。
- `eva-cli` 和 `eva-runtime` 仍主要是边界与占位入口，尚未形成可运行的最小 Runtime
  闭环。

因此，当前定位不再是“准备创建模块结构”，而是：

1. 巩固已拆分 crate 的契约边界；
2. 把 manifest、policy、Lua host API、EventBus、Scheduler、Agent lifecycle 等占位模块补成可编译、可测试契约；
3. 将 CLI、配置校验、runtime builder 和最小示例串成第一条可运行链路。

## 阶段总览

| 阶段 | 状态 | 当前结论 |
| --- | --- | --- |
| 1. 架构和方案文档 | 基本完成 | 中文详案、英文入口、官网与 wiki 已能支撑第一轮实现。 |
| 2. 模块划分 | 基本完成 | Rust workspace 和主要 crate 边界已落地，后续只应按验证需要微调。 |
| 3. 契约定义 | 进行中 | `eva-core` 已有第一批基础契约；manifest、policy、Lua host API 和跨 crate trait 仍需补齐。 |
| 4. 最小可运行骨架 | 部分完成 | CLI shim 和 crate 边界存在，但 `eva run`、`eva validate`、`eva doctor` 还未形成真实命令面。 |
| 5. 最小端到端 Runtime 闭环 | 未完成 | 这是下一硬里程碑，需要以 `examples/basic/` 和集成测试证明。 |
| 6. 逐模块实现 | 未系统展开 | 应在最小闭环跑通后逐个模块补真实行为。 |
| 7. 集成、加固和发布准备 | 未开始 | 等核心闭环和用户命令稳定后进入。 |

## 进度阶段

### 1. 架构和方案文档

目的：

- 明确产品目标、非目标、系统边界和核心假设。
- 描述运行时模型、扩展模型、记忆模型、发现模型、硬件接入、配置系统和恢复策略。
- 在实现前记录设计风险。

已落地：

- 总体架构、模块划分、Runtime/EventBus/Scheduler/Agent/Adapter/Memory/Discovery/Config/
  Hardware、热更新、备份迁移、进程级升级、CLI 命令面等中文详案。
- 英文默认公开入口和摘要型英文文档。
- 官网静态站点、文档索引、博客内容和 i18n manifest。
- `docs/zh-CN/README.md` 中的推荐阅读顺序和内容权威说明。

仍需维护：

- 当实现偏离方案时，先更新对应中文详案，再同步英文摘要或全文。
- 风险评审需要随着最小闭环实现继续更新，不能停留在设计期假设。

### 2. 模块划分

目的：

- 把架构转换成具体源码结构。
- 用目录、crate 和模块名称表达职责边界。
- 在不急着写完整逻辑的前提下，让系统形态先落地。

已落地：

- 根目录 `Cargo.toml` 已定义 Rust workspace。
- 主要 crate 边界已经存在：
  - `eva-core`
  - `eva-config`
  - `eva-policy`
  - `eva-observability`
  - `eva-storage`
  - `eva-eventbus`
  - `eva-scheduler`
  - `eva-agent`
  - `eva-lua-host`
  - `eva-capability`
  - `eva-adapter`
  - `eva-mcp`
  - `eva-discovery`
  - `eva-memory`
  - `eva-hardware`
  - `eva-backup`
  - `eva-lifecycle`
  - `eva-runtime`
  - `eva-cli`
- 根入口 `src/main.rs` 已通过 `eva_cli::run()` 连接到 CLI crate。

仍需收敛：

- `examples/basic/` 尚未成为第一条可运行链路。
- 部分 crate 目前只有职责常量或占位模块，不能被视为行为完成。
- 后续不应继续扩张 crate 数量，除非有清晰的编译边界、依赖方向或测试收益。

### 3. 契约定义

目的：

- 在填充业务逻辑前，先定义接口和数据契约。
- 让模块边界变得可测试、可评审。
- 避免运行时代码硬编码尚未稳定的文档假设。

已落地：

- `eva-core` 已定义并导出基础契约：
  - Topic 与 TopicPattern。
  - Event、EventPayload、EventTarget、TraceContext、EventMetadata。
  - AgentId、AdapterId、CapabilityId、EventId、GenerationId、RequestId 等 ID newtype。
  - CapabilityName、CapabilityRef、ProviderHint。
  - InvokeRequest、InvokeResponse、InvokeInput、InvokeOutput、InvokeStatus。
  - EvaError、ErrorKind、ProviderCode、ErrorContext。
- `config/schemas/` 已有第一批 JSON Schema 文件：
  - `agent.schema.json`
  - `adapter.schema.json`
  - `capability.schema.json`
  - `eva.schema.json`
  - `policy.schema.json`
  - `routes.schema.json`
- `crates/eva-core` 已有针对 Topic、Event、Invoke、ID 和 Error 的单元测试。

下一步必须补齐：

- `AgentManifest`、`AdapterManifest`、`CapabilityManifest` 的 Rust 类型、反序列化和 schema 对齐测试。
- MCP、hardware、sandbox、adapter、workspace policy 的有效配置类型和合并规则。
- EventBus trait、Subscription、DeadLetter、RecoverableEventLog 的最小可测试契约。
- Scheduler route rule、match result、queue target 和冲突处理契约。
- Agent lifecycle、mailbox、runtime state 和隔离执行边界。
- Lua host API 契约：
  - `ctx.tools`
  - `ctx.host`
  - `ctx.memory`
  - `ctx.global_memory`
  - `ctx.knowledge`
- Capability 命名、provider 选择、冲突处理和审计字段。

完成标准：

- 每个公开契约都有编译期测试、单元测试或 schema 校验测试。
- 文档中的字段名、Rust 类型和 JSON Schema 不出现三套命名。
- 占位 `RESPONSIBILITY` 常量不再被误认为已实现行为。

### 4. 最小可运行骨架

目的：

- 证明仓库可以构建和启动。
- 在深入实现 Runtime 之前，先建立开发闭环。

已落地：

- 根二进制入口已存在。
- `eva-cli` crate 已承接用户入口职责。
- `eva-runtime` crate 已作为组合根边界存在。
- `.github/workflows/pages.yml` 已覆盖网站发布链路。

仍需实现：

- 真实 CLI 命令面：
  - `eva run`
  - `eva validate`
  - `eva doctor`
  - `eva inspect`
- 配置文件读取、schema 校验和错误输出。
- Runtime builder 的最小装配流程。
- 结构化日志、trace 和 audit 的最小输出格式。
- no-op 或 in-memory 实现，用于让命令在没有真实外部 adapter 时也能运行。
- CI 中的 Rust format、lint、test 和 schema validation。

完成标准：

- 从干净 checkout 可以构建 CLI。
- `eva doctor` 可以报告本地环境、配置路径、schema 状态和已发现能力数量。
- `eva validate` 可以校验最小项目配置、manifest 和 policy。
- `eva run --example basic` 或等价命令可以启动最小 runtime skeleton。
- Rust 测试和文档/网站校验在 CI 中运行。

### 5. 最小端到端 Runtime 闭环

目的：

- 把骨架变成最小真实 Eva-CLI 行为。
- 在扩展平台能力前，先用一条很窄的可运行路径验证核心架构。

目标闭环：

1. 用户输入进入 CLI/Ingress。
2. Ingress 发布 typed Topic event。
3. EventBus 接收并持有事件。
4. Scheduler 按 TopicPattern 把事件路由到一个 Agent queue。
5. AgentRuntime 从 queue 中取出事件。
6. Lua Agent 在隔离状态中处理事件。
7. Lua Agent 调用一个受控 Rust capability/tool。
8. Rust 在执行前校验 schema 和 policy。
9. Tool 结果以结构化值返回 Lua。
10. Runtime 输出 trace 和 audit 数据。
11. 失败时返回结构化、带 retry 语义的错误。

完成标准：

- `examples/basic/` 可以运行。
- 一个 Lua Agent 可以处理一个事件。
- 一个受控 tool 调用可以通过 Rust 校验并执行。
- trace、audit 和结构化错误可观察。
- 回归测试覆盖完整闭环。
- 文档中的 quickstart 与真实命令一致。

### 6. 逐个模块补齐具体实现

目的：

- 在最小闭环验证架构后，再按模块补齐真实行为。
- 每个模块实现都应基于契约和测试。

推荐顺序：

1. Config 和 schema 校验。
2. EventBus。
3. Scheduler。
4. Agent mailbox 与 lifecycle。
5. Lua runtime 和 host bindings。
6. Capability/tool layer 和 AdapterRegistry。
7. Error model、trace 和 audit。
8. MemoryService 和 ContextBuilder。
9. Knowledge 访问。
10. Discovery。
11. Hot reload 和 generation switching。
12. MCP 和 Skill adapter。
13. HardwareAdapter。
14. Backup、restore、migration package 和 ReleaseSnapshot 支持。
15. Process lifecycle、draining、rollback 和 supervisor 集成。

建议：

- 一次只实现一个模块切片。
- 优先删除和简化契约，而不是继续增加层级。
- 每个模块需要有聚焦测试，才能视为完成。
- 跨模块行为应通过 examples 和集成测试验证，不能只靠单元测试。
- 共享类型必须优先沉到 `eva-core` 或对应边界 crate，避免 runtime 反向定义底层契约。

### 7. 集成、加固和发布准备

目的：

- 从内部可运行推进到可发布 CLI。

预期交付物：

- 已支持场景的可运行 examples。
- 面向用户的 quickstart。
- 安装说明。
- 覆盖 format、lint、unit test、integration test、schema validation 和 website build 的 CI。
- macOS、Linux、Windows 跨平台检查。
- 针对 sandbox、policy、secret、文件写入、MCP 和硬件访问的安全评审。
- 版本化 release notes。
- 破坏性变更迁移说明。

完成标准：

- 新用户可以安装 Eva-CLI，跑通 quickstart，并诊断常见环境问题。
- 文档示例和真实行为一致。
- 已知限制被明确写出。
- 不支持的高级能力在 CLI 输出、文档和 release notes 中都明确标记。

## 1.0 发布标准

Eva-CLI 1.0 不代表所有设想都已经实现，而是代表核心承诺已经稳定。

1.0 必须满足：

- CLI 安装和启动可靠。
- `eva doctor`、`eva validate` 和最小 `eva run` 链路稳定。
- 最小 Agent runtime 闭环稳定。
- Manifest、policy 和 Lua host API 契约已版本化。
- Lua Agent 可以通过 Rust 校验调用受控 tool/capability。
- 结构化错误、trace 和 audit 可用。
- Config validation 能尽早拦截不支持或不安全的配置。
- EventBus、Scheduler、AgentRuntime、AdapterRegistry 和 Runtime builder 的最小行为有集成测试。
- 文档、示例和 release artifact 与实现一致。
- 破坏性变更有迁移说明。
- 尚不支持的高级能力被明确标记。

1.0 不强制包含：

- 完整 MCP 双向生态。
- 全量硬件接入。
- 完整备份/迁移/ReleaseSnapshot 自动化。
- 复杂多 Agent 策略优化。
- IDE 插件和移动端入口。

这些能力可以进入 1.x 路线，但不能阻塞核心 runtime 闭环的稳定发布。

## 当前最近里程碑

最近里程碑应聚焦在“让现有 workspace 从边界骨架变成最小可运行系统”：

1. 补齐 manifest、policy 和 routes schema 的 Rust 类型与校验测试。
2. 为 `eva validate` 实现最小配置加载和 schema 校验。
3. 为 `eva doctor` 输出本地环境、配置路径、schema 和 discovery 状态。
4. 为 EventBus 和 Scheduler 定义可测试 trait 与 in-memory 实现。
5. 为 AgentRuntime 和 Lua host bindings 定义最小处理路径。
6. 建立 `examples/basic/`，覆盖“事件进入、路由、Agent 处理、受控 capability 返回结果”的闭环。
7. 将该闭环加入集成测试和 CI。

这样的节奏能把项目从“架构已经拆好”推进到“核心承诺可以运行”，同时避免继续堆叠尚未被端到端验证的抽象。
