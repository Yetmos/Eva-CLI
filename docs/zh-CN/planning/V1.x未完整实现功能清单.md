# V1.x 未完整实现功能清单

更新时间：2026-07-07

本文记录 Eva-CLI `V1.7.2-alpha` 中已经具备部分实现、但尚未进入完整真实执行路径的功能。它补充 [当前项目从零到完整实现实施计划](当前项目从零到完整实现实施计划.md)，用于区分“已实现的契约、诊断、计划和受控 envelope”与“仍待补齐的 runtime 能力”。

当前结论：V1.7.2-alpha 已经形成可编译 workspace、稳定 CLI 输出、in-memory basic runtime、外部能力诊断、记忆上下文、硬件绑定计划、备份/生命周期计划、发布加固门禁、第一版 durable backend 布局/迁移锁基线、durable EventBus redrive 基线、durable task snapshot、durable audit sink、filesystem artifact metadata hardening、runtime recovery scanner、event redrive recovery checkpoint、recovery audit smoke、通过 `inspect durable` 暴露的 durable backend diagnostics、受限 Lua VM `on_event` 真实执行边界、只读 Lua context table、Lua host log/audit observation，以及受控 `ctx.tools.call` capability binding。但它仍是 alpha 检查点，不是带完整 runtime audit wiring、真实 provider、真实硬件、真实 apply gate、Lua 资源限制和签名安装器的完整 runtime 发行版。

## 状态口径

| 状态 | 含义 |
| --- | --- |
| 已有边界 | 已有类型、trait、命令、诊断或 plan-first 输出，可以作为后续实现基础。 |
| 部分实现 | 已可在受控条件下运行，但只覆盖 in-memory、mock、simulated、dry-run 或 envelope 路径。 |
| 未完整实现 | 尚未接入真实副作用、持久化、进程管理、外部协议、OS 权限、签名发布或完整策略门禁。 |
| 下一步 | 将现有边界升级为可审计、可回滚、可测试的真实执行路径。 |

## 计划到执行的升级顺序

| 阶段 | 建议切片 | 目标 | 退出标准 |
| --- | --- | --- | --- |
| P0-A | Durable storage + runtime crash recovery | 先让 EventBus、task、audit、artifact 具备可恢复后端 | 重启后可继续查询 task/event/audit；`cargo test --workspace` 和 durable 集成测试通过 |
| P0-B | 真实 Lua VM + host API | 从 V1.7.1 真实执行边界继续补资源限制和 host API | Lua sandbox、timeout、memory limit、host API 权限测试通过 |
| P0-C | Adapter/MCP 真实 provider | 启用 stdio/http/MCP provider，但保留 manifest、policy、audit、timeout 边界 | 未授权 provider 不启动；超时/限流/输出限制/脱敏测试通过 |
| P0-D | Restore/Upgrade apply gate | 从 plan-only 进入可确认、可回滚的真实 apply | 签名 artifact、backup evidence、apply lock、health check、rollback 测试通过 |
| P1-A | Hardware real driver path | 接入真实设备前先固定 simulator 回归和 OS 权限模型 | 真实 driver 不暴露 raw handle；hardware simulator CI 可重复 |
| P1-B | Policy/schema/observability 深化 | 将配置、权限和审计从基础契约升级为运行时门禁 | schema validator、policy domain parser、persistent audit sink 通过集成测试 |
| P1-C | Release artifact 分发 | 补签名、安装器、包管理器和 provenance | release check 能验证 artifact 证据并阻断缺失项 |

## 当前可验证证据

| 验证命令 | 当前预期 |
| --- | --- |
| `cargo run -- --version` | 输出 `eva 1.7.2-alpha`、`V1.7.2-alpha`、`lua_vm_execution_v1.7.1` 和 `lua_host_bindings_v1.7.2`。 |
| `cargo run -- restore plan --output json` | 输出 `apply_allowed:false`，不执行恢复。 |
| `cargo run -- hardware bind --adapter scale-main --apply --output json` | 输出 `status:"blocked"`、`handle_granted:false`，不打开真实设备。 |
| `cargo run -- release check --output json` | `status:"ready"`，但保留 restore/upgrade apply 的 warning gate。 |
| `cargo test --workspace` | 作为当前所有已实现边界的回归基线。 |

## 维护规则

| 规则 | 说明 |
| --- | --- |
| 先测试后扩大副作用 | 从 diagnostic/envelope/plan 升级到真实执行前，先补失败用例和回归测试。 |
| 保持 plan-first | restore、upgrade、hardware、release pointer、权限扩大等高风险路径必须先输出计划和风险。 |
| 不绕过边界 | Lua、Agent、Discovery 不直接拿 provider、设备、文件、socket 或进程句柄。 |
| 文档与实现同步 | 任何真实执行路径落地后，需要同步更新本清单、实施计划、使用手册和 release 文档。 |
