# V1.x 未完整实现功能清单

更新时间：2026-07-09

本文记录 Eva-CLI `V1.11.5-alpha` 中已经具备部分实现、但尚未进入完整真实执行路径的功能。它补充 [当前项目从零到完整实现实施计划](当前项目从零到完整实现实施计划.md)，用于区分“已实现的契约、诊断、计划和受控 envelope”与“仍待补齐的 runtime 能力”。

V1.11.5.1 更新：`emit` 不再只是边界保留命令；CLI 已可构造 typed `eva-core::Event`，并向 in-memory 或 `--durable-backend` EventBus 发布，返回稳定 text/JSON receipt。后续缺口收窄为 `agent` 与 `capability` 命令面、常驻 runtime 消费路径和更完整 operator UX。
V1.11.5.2 更新：`agent status/drain/reload` 已可作为 CLI evidence 边界运行；它从 ProjectConfig manifest、`AgentRuntime`、`AgentLifecycle`、`DrainCoordinator` 和 `GenerationController` 输出 lifecycle、drain plan 与 generation reload evidence，并在无 daemon 时显式保持 `mutation_executed:false`。V1.12.5 已把有 daemon 时的 drain/reload mutation 接入 daemon state，后续缺口转为完整 provider restart/apply。
V1.11.5.3 更新：`capability list/probe/call` 已可作为 CLI operator 边界运行；它复用 `CapabilityRegistry`、provider selection plan、`CapabilityPermissionGate`、`RuntimePolicyGate`、`CapabilityRouter` 和 `AdapterBackedCapabilityHost` 输出 provider plan、permission gate、runtime policy 与 invoke response evidence。`call` 默认 dry-run，只有 `--confirm <request-id>` 匹配后才执行受控 invoke；provider 超出 manifest allowlist 会返回 policy error。
V1.12.1 更新：`daemon start/status/stop` 已可作为本机 daemon 进程边界 smoke 运行；`start` 会创建并验证 pid/lock/state 路径、durable backend、policy domain 和 observability JSONL sink，输出 foreground/dev mode 与 shutdown contract evidence，并显式保持 `provider_processes_started:false`。重复启动会在 lock 文件上返回 conflict；坏 durable backend 不会写入 daemon state。
V1.12.2 更新：显式 `daemon start --no-shutdown-after-smoke` 会保持前台 control loop，CLI `daemon status/shutdown/submit/cancel/drain/reload` 通过受控 filesystem mailbox 发送 request 并读取 response；无 running daemon 时返回稳定 `unavailable`，request/response 均带 `trace_id`、`request_file` 和 `response_file` evidence。
V1.12.3 更新：`TaskStateSnapshot` 已升级为 durable task lifecycle，支持 queued/running/cancelling/timed_out/completed/failed/interrupted/recovering、heartbeat、deadline、cancel token、interrupted reason 和日志追加；daemon submit/cancel 已写入 queued/cancelling；recovery 已覆盖 cancelling；`AgentRunControl` 可携带 cancel token/deadline evidence。
V1.12.4 更新：daemon foreground control loop 已接入 scheduler retry tick；`DurableEventBus` 可查询 due dead-letter、event log status 和 latest replay record；due replay event 会投递到 scheduler mailbox，并以 `scheduler-retry` consumer 写 durable log ack/fail；recovery checkpoint 和 `inspect durable` 已跳过存在 replay 子事件的 dead-letter，避免 live tick 与重启恢复重复 redrive。
V1.12.5 更新：`agent drain/reload` 在发现 running daemon 时会通过 filesystem mailbox 执行真实 daemon-side mutation；daemon 写入 `agent-control.state`，记录 drain `accepts_new_work:false`、reload 后 `selected_generation_for_new_work` 和旧 generation `draining` 状态。无 daemon 时 CLI 继续保留 `mutation_executed:false` evidence path，不破坏原 JSON contract。
V1.12.6 更新：`release check` 已新增 `REL-DAEMON-RUNTIME-001` required gate，汇总 V1.12 daemon process boundary、filesystem mailbox、durable task lifecycle、scheduler retry tick 和 daemon-backed agent drain/reload mutation evidence；CI 也显式运行 daemon control、retry、drain/reload smoke。该 gate 只证明本机 foreground/filesystem daemon readiness，不等于生产 OS service manager 或完整 provider supervision。

## 状态口径

| 状态 | 含义 |
| --- | --- |
| 已有边界 | 已有类型、trait、命令、诊断或 plan-first 输出，可以作为后续实现基础。 |
| 部分实现 | 已可在受控条件下运行，但只覆盖 in-memory、mock、simulated、dry-run 或 envelope 路径。 |
| 未完整实现 | 尚未接入真实副作用、持久化、进程管理、外部协议、OS 权限、签名发布或完整策略门禁。 |
| 下一步 | 将现有边界升级为可审计、可回滚、可测试的真实执行路径。 |

### 被阻塞功能

| 功能 | 阻塞原因 | 当前证据/行为 | 解除条件 |
| --- | --- | --- | --- |
| destructive restore apply | 高风险文件 mutation 不能只靠 CLI 参数触发 | `restore plan` 输出 `apply_allowed:false`；`restore apply` 通过 gate 后仍保持 `mutation_executed:false` | 保留 signed archive、pre-restore backup、policy approval、filesystem lock、health gate 和 rollback 测试后，再接入真实文件 mutation |
| daemonized supervisor handoff | 目前只有本地 state-store handoff，不具备生产服务管理器 | release check 的 `SEC-LIFE-001` 是 warning gate；文档要求替换 local supervisor adapter smoke | 实现 OS service-manager adapter、真实 runtime binary lifecycle 和失败 rollback |
| 硬件 raw I/O | 默认样例硬件 manifest disabled，且 CLI 不授予 raw handle | `hardware bind --apply` 返回 `status:"blocked"`、`handle_granted:false`，风险中注明不打开 raw I/O | 真实 driver、OS permission provider、设备模拟回归和审计 gate 全部落地 |
| 未授权 provider/capability 调用 | provider 必须来自 manifest allowlist，且通过 capability permission/runtime policy | `capability call` 默认 dry-run；confirm 不匹配或 provider 超 allowlist 会在 invoke 前失败 | provider supervision、credential/session policy 和 runtime audit 后端完善后再扩大执行范围 |
| MCP 未列入 allowlist 的 tool | MCP 工具发现不等于执行授权 | `mcp probe` 对未列入 allowlist 的 tool 返回 blocked 诊断 | 保持 allowlist/policy/audit 后，按 adapter manifest 显式授权 |
| 生产发布上传 | release evidence gate 只验证证据，不持有生产签名和仓库发布凭据 | `release check` 可验证 artifact/distribution/scanner/benchmark evidence，但不执行仓库上传 | 接入生产 signing/attestation credential 和 OS 包仓库发布流程 |

### 已知缺陷和风险

| 类型 | 项目 | 现象 | 影响 | 建议 |
| --- | --- | --- | --- | --- |
| 已验证测试状态 | 当前扫描未发现生产 `TODO/FIXME/unimplemented!` 标记；`cargo run -- --version` 和 `cargo run -- release check --output json` 通过 | 只发现 `crates/eva-adapter/src/runtime.rs` 中测试 HTTP mock 的 `panic!`，不属于生产路径 | 暂无已复现的编译失败或命令失败 | 后续仍需在功能改动后跑 `cargo test --workspace` |

| UX 语义风险 | `restore apply` | gate 可返回 `apply_allowed:true`，但真实文件恢复仍未执行 | 使用者可能误把 gate 通过理解为恢复已完成 | 在输出和文档中继续保留 `mutation_executed:false`，真实 mutation 前增加更强确认文案 |

| 架构缺口 | runtime recovery | 可恢复 task/event/audit 证据，但不能恢复 provider 进程执行态 | 进程崩溃后的外部 provider 调用无法完整续跑 | P0-A/P0-C 阶段补 provider supervision 与执行态恢复 |
| 观测缺口 | observability | file JSONL 已有，但 runtime/provider/task 生产事件未全量接入 OpenTelemetry/metrics exporter | 生产部署缺少统一 trace/metrics 后端 | 先接完整 runtime audit wiring，再接 OTel SDK/exporter 和 retention/rotation |
| 文档维护风险 | 中英文同步 | 中文为详细事实源，英文只适合摘要入口 | 英文页若继续写 full-detail，容易滞后并产生第二事实源 | 英文页保持 summary 状态，细节以本文为准 |
