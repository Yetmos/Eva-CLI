# V1.x 生产能力阻塞项

> 更新：2026-07-18  
> 依据：[V1.x 未完整实现功能清单](V1.x未完整实现功能清单.md) 与 [V1.x 真实运行时能力补齐实施计划](V1.x真实运行时能力补齐实施计划.md)

## 1. 文档目的

本文只记录会阻塞 V1.x 生产能力交付或生产闭环判定的事项，并说明解除路径。`release check` 返回 `ready` 仅表示当前输入没有被标记为 blocked，不代表生产就绪。当前公开版本仍为 `1.11.5-alpha`；代码中的 V1.12-V1.17 标识是兼容性 evidence ID，不是已发布版本。

阻塞项分为：

- **实现阻塞**：仓库中仍缺少生产运行时或平台集成代码。
- **环境阻塞**：需要平台、硬件、数据库、凭据或仓库权限才能实施和验收。
- **证据阻塞**：实现存在，但没有可信的真实执行证据，不能满足生产 gate。

## 2. 总体阻塞摘要

| 优先级 | 阻塞范围 | 当前状态 | 解除原则 |
| --- | --- | --- | --- |
| P0 | W2-W5：平台服务、Provider 监管、MCP 生产传输、备份恢复与升级 | Open/Blocked | 先完成 W1 的后台运行时基础，再按依赖顺序接入真实 OS、进程和数据面 |
| P0 | W10：生产闭环 gate | Not started/Blocked | W0-W9 的声明范围、执行证据、覆盖率和 provenance 全部可验证后才能启动 |
| P1 | W6-W9：配置发现、硬件、记忆/可观测性、分发 | Open/Blocked | 本地无凭据 dry-run 与代码实现先行；外部环境到位后补真实证据 |
| 外部 | EXT-04/05/06/07/09 | 未满足或未决策 | 明确责任人、环境、凭据保管和验收窗口，形成可引用的决策记录 |

## 3. P0 阻塞项与解决方案

### B-P0-01 平台 Service Manager 未完成（W2）

**阻塞原因**：目前只有 trait 和 fake Adapter，缺少 Windows Service、systemd、launchd 的真实 install/status/start/stop/restart、开机恢复和权限错误验证。

**解决路径**：

1. 固化 service identity、state/pointer 和升级回滚协议，复用 W1 的 lease、heartbeat 和恢复分类。
2. 分别实现三个 OS Adapter，所有命令使用结构化参数，不通过 shell 拼接。
3. 在受控 Windows/Linux/macOS runner 上执行安装、崩溃、重启、重启后恢复、stale-lock 和 rollback 集成测试。
4. 产出绑定 commit、平台、服务版本、执行者和时间戳的 evidence；fake Adapter 只能作为单元测试证据。

**解除标准**：三平台真实 service 生命周期和失败回滚测试通过，权限失败可预测且无孤儿进程。

### B-P0-02 Provider 进程监管与凭据隔离未完成（W3）

**阻塞原因**：单次 runner 已有 PID/清理/重试基础，但缺少 daemon-owned 长驻监管、跨进程 admission、用户隔离、credential vault，以及 Linux/macOS crash-loop/daemon-kill 证据。

**解决路径**：

1. 将 provider process table、daemon generation 和 service identity 纳入同一 ownership 协议。
2. 实现跨平台 process-group/Job 清理、重启退避、crash-loop 熔断和 stale generation 回收。
3. 用 OS secret store/KMS 提供 credential provider；禁止 secret 进入日志、JSON 输出和 artifact。
4. 增加强制 kill、daemon 重启、并发 admission、用户隔离和敏感信息扫描测试。

**解除标准**：故障后无 orphan process，重试策略可观测，凭据零泄露，跨进程限制有效。

### B-P0-03 MCP 生产传输未完成（W4）

**阻塞原因**：现有 stdio 和明文 HTTP/JSON-RPC 主要由 fixture 覆盖，缺少 HTTPS/TLS、Streamable HTTP/SSE、stream abort 资源回收和真实外部 server 兼容矩阵。

**解决路径**：

1. 在 W3 监管边界内实现 TLS client、证书校验、认证和超时/取消传播。
2. 实现 streaming session 生命周期、背压、断流清理和输出上限。
3. 选定至少一组外部 MCP server，建立版本、认证、工具调用和异常场景矩阵。
4. 将真实运行结果写入 evidence；移除 all-true 静态 fixture 对生产 gate 的替代资格。

**解除标准**：TLS/auth 错误稳定、断流无资源泄漏、外部 server 实跑通过。

### B-P0-04 备份恢复与升级交接不具备生产信任边界（W5）

**阻塞原因**：当前主要是本地、合成或内存 snapshot；缺少真实数据选择、持久 catalog、KMS 签名/加密、可信 plan、target-root 限制、远程灾备及 blue-green rollback。

**解决路径**：

1. 先确定数据范围、快照 catalog schema、KMS/HSM、密钥轮换/撤销和 custody policy。
2. 为 restore/upgrade plan 增加 provenance、签名校验、目标根目录 allowlist 和 operator confirmation。
3. 实现真实 artifact 与 service-manager 的 handoff、drain、pointer 更新和失败回滚。
4. 进行真实 dataset restore、候选版本失败、重启恢复和 pointer 不漂移演练。

**解除标准**：未签名/越权 plan 被拒绝，失败 restore/upgrade 可自动或人工回滚且数据和 pointer 一致。

### B-P0-05 生产证据闭环不足（W0/W10）

**阻塞原因**：静态 declaration、fixture 和真实 measurement 尚未完全分离；缺少可信 executor、freshness、scope coverage 和跨证据 subject 一致性。

**解决路径**：

1. W0 统一捕获 argv、退出结果、日志摘要、artifact digest、commit、环境、executor 和时间戳。
2. W10 实现 versioned ProductionScope、closure manifest、trust/freshness/kind verifier 和 coverage verifier。
3. 生产 profile 强制拒绝 static declaration、过期 evidence、未覆盖 scope 和不受信 executor。
4. 增加成功/失败 E2E，确保 gate 输出 remediation 而不是静态 pass。

**解除标准**：每个声明的生产能力都有可重放的真实 evidence，gate 可机器判定发布或阻断。

## 4. P1 阻塞项与解决方案

| 编号 | 阻塞项 | 解决方案 | 解除标准 |
| --- | --- | --- | --- |
| B-P1-01 | 配置/Discovery 仍缺 watcher、原子 live swap、真实 registry/auth、持久 TTL cache（W6） | 先固定 schema/denial contract；实现 generation watcher、超时可取消 source、PATH/registry client 和原子 cache；补并发 reload、损坏 cache、权限拒绝测试 | reload 可恢复且不产生半配置；每个 mutation 有显式 policy decision |
| B-P1-02 | 硬件只有 simulator，缺 OS permission、USB/serial、hotplug、I/O 和 fixture（W7） | 提供命名的真实或虚拟 fixture；先做 USB/serial baseline，再做 BLE/socket/vendor；raw handle 仅留在硬件边界 | 覆盖 permission denial、I/O、reconnect、hotplug、lease cleanup，且无 raw-handle escape |
| B-P1-03 | Memory/Knowledge 与 Observability 缺长驻 worker、真实 retrieval、OTel flush、database sink、retention（W8） | 先决定数据库、schema、凭据和 retention policy；接入 restart-safe schedule、adapter-backed retrieval、OTel lifecycle、降级队列和后台 retention | 不写 demo seed；重启可恢复；collector/db 故障按策略降级并产生 retention/load evidence |
| B-P1-04 | Distribution 缺签名/公证、attestation、仓库上传和上传后安装验证（W9） | 先完成无凭据 metadata/validator dry-run；再接入 Windows/macOS/Linux signer、SBOM/attestation、上传事务和 clean runner install smoke | 产物可验证签名和 digest；上传失败不报告完成；公开下载可干净安装 |

## 5. 外部前置条件（EXT）

| ID | 需要提供 | 解决动作 | 责任输出 |
| --- | --- | --- | --- |
| EXT-04 | Windows/Linux/macOS service host；真实或虚拟硬件 fixture | 由 CI/运维建立受控 runner，固定版本、权限和清理策略 | 平台服务与硬件 integration evidence |
| EXT-05 | 数据库选型、schema、凭据、retention/运维策略 | 完成架构决策记录，再实现 sink、migration 和 worker | database/retention evidence |
| EXT-06 | CI OIDC、KMS/HSM、代码签名/公证身份、密钥轮换策略 | 使用短期凭据和最小权限，禁止把密钥写入仓库或日志 | signing、notarization、attestation evidence |
| EXT-07 | Homebrew/Winget/Apt/GHCR 仓库所有权和 publish token | 先 dry-run，后采用可重试、幂等上传事务并做下载校验 | repository upload/install evidence |
| EXT-09 | 生产 collector/database/负载环境 | 固定采样、保留期、cardinality 和降级预算 | telemetry、retention、load evidence |

外部条件未到位时，代码工作不得标记为完成；可以继续实现无凭据 validator、fake/fixture 单测和本地 dry-run，但这些不能替代生产证据。

## 6. 推荐解除顺序

1. 完成 W0 证据协议和 W1 后台运行时基座（已有实现需完成验证）。
2. 并行推进 W2/W3；平台 service host 到位后立即运行真实生命周期测试。
3. 在 W3 稳定后完成 W4；W2、W1、W0 和 KMS/plan policy 到位后完成 W5。
4. W6-W9 按各自依赖并行；凭据只阻塞签名/上传阶段，不阻塞本地实现和 dry-run。
5. 所有 in-scope 项目有真实 evidence 后实现并启用 W10 production closure gate。

## 7. 通用验收与验证

每个阻塞项解除时必须同时提供目标行为、兼容性契约、负向测试和环境 evidence，并保持以下基线通过：

```powershell
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
./scripts/validate-cli-json-contracts.ps1
./scripts/build-site-i18n.ps1
./scripts/validate-i18n.ps1
./scripts/validate-version-management.ps1
cargo run -q -- release check --output json
```

生产闭环还必须能证明：commit/artifact digest、执行环境、执行者、时间戳、scope coverage、gate 结果和 remediation 均可追溯。未满足这些条件前，版本保持 alpha，不得将内部 evidence ID 宣称为已发布版本。
