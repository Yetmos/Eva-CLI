## 1. 文档目的

本文只记录会阻塞 V1.x 生产能力交付或生产闭环判定的事项，并给出解除路径。`release check` 返回 `ready` 仅表示默认 alpha 输入没有 required gate 被标为 blocked，不代表生产就绪。当前公开版本仍为 `1.11.5-alpha`；代码中的 V1.12-V1.17 标识是兼容性 evidence ID，不是已发布版本。

阻塞项分为：

- **实现阻塞**：仓库中仍缺少生产运行时、平台集成或闭环代码。
- **环境阻塞**：需要平台、硬件、数据库、凭据、外部服务或仓库权限才能实施和验收。
- **证据阻塞**：代码存在，但缺少可信真实执行 evidence，不能进入生产 gate。
- **决策阻塞**：生产范围、协议、信任根或运维策略尚未批准，继续实现会产生不稳定接口。

当前实施计划共有 123 个逻辑项：0 个可开始、56 个已完成、12 个验证中、55 个已阻塞。W0 与 W1 已 Complete；其余 Work Package 的阻塞必须按逻辑 ID 和 EXT/DEC 前置项跟踪。

**阻塞原因**：W2-L01/L02/L06/L07 已完成，Windows Service、systemd、launchd Adapter 代码已实现并处于验证中；canonical hidden entrypoint、direct/no-respawn daemon ownership、PID/lease identity、stop-token drain/shutdown 和 drift 拒绝均有仓库内回归。当前仍缺少受控提权 Windows、PID1 systemd Linux、macOS system/GUI host 的真实生命周期 transcript；W2-L08 boot/restart recovery、W2-L09 destructive harness 和 W2-L10 production evidence gate 仍被该环境证据边界阻塞。

**解决路径**：

1. 在 EXT-01 host 上完成 W2-L03 至 L05 的 Windows create/start/restart/stop/delete、Linux root/non-root、macOS system/GUI 生命周期 transcript。
2. 以已完成的 W2-L07 direct entrypoint、service identity 和 W1 lease/recovery 为前置，完成 W2-L08 boot/restart recovery 与 queued-work/no-double-worker 验收。
3. 由 W2-L09 执行三平台 reboot/crash/destructive lifecycle，并以 finally cleanup 失败作为 job 失败条件。
4. 由 W2-L10 回读三平台 transcript，绑定 commit、平台、service identity、executor、时间戳和结果 digest；Fake 或本地 fixture 不得通过 production gate。

**解除标准**：三平台真实 service 生命周期、权限拒绝、崩溃/重启恢复和 finally cleanup 均通过；fake Adapter 不能满足 production evidence。

### B-P0-02 Provider 用户隔离、凭据和长驻所有权未完成（W3）

**阻塞原因**：W3-L01 至 L07 已完成，跨进程 admission 不再是缺口；W2-L07 service identity 前置也已就绪。W3-L08 的代码边界已接线并进入验证中，但 EXT-01 原生 identity evidence 尚未完成：Unix helper 和 Windows same-service-token 回归已覆盖，仍缺受控 Linux/macOS/Windows service host 的真实 uid/gid 与 token 隔离 transcript；macOS 显式切换当前未开放，等待受控 no-suid 或 sandbox 方案，`Current` 仍可用。当前 Windows 只允许与 daemon primary token 同 SID 的账户继承 token，不提供任意跨账户 `CreateProcessAsUserW`/`CreateProcessWithTokenW` 切换；代码对检测到的 non-root supplementary groups fail closed，但仍缺 host 对组状态的 attestation，root 创建目录的 UID/GID handoff 也未由 host policy 证明。W3-L09 已接入 vault/session trait、显式 Memory 测试 vault 和生产 fail-closed default：stdio/MCP stdio/Skill 使用最小环境，HTTP/MCP HTTP 从 session 解析凭据；Skill workspace/artifact 目录和最终文件已做 no-follow、owner/regular-file 检查与句柄读取，且 transport/report Debug、audit、artifact、stream、error 均有脱敏边界。该项仍处于验证中，因为仓库尚未接入真实 OS secret store/KMS authority，也缺 EXT-01 原生 identity、跨 UID ownership、目录 ancestor race/完整 TOCTOU 与生产 leak-scan evidence。W3-L10 daemon-owned supervisor/drain 已接线并处于验证中；仍需验证其 renew/restart/shutdown/generation recovery 与 native evidence，W3-L11 负责三平台 production evidence；Linux/macOS 原生 crash-loop/daemon-kill evidence 尚未齐全。

**解决路径**：

1. 在 EXT-01 受控 host 回读 W3-L08：完成 Unix uid/gid、supplementary-group、目录 ownership 与 Windows same-SID service-token 的原生 transcript；macOS 显式切换当前未开放，等待受控 no-suid 或 sandbox 方案，`Current` 仍可用；禁止身份提升并验证跨账户访问隔离，未知或不同 SID 在 spawn 前拒绝。
2. 接入 OS secret store/KMS-backed vault session，secret 只通过引用进入 provider，禁止进入日志、JSON 和 artifact；当前仓库已接入 session trait、Memory 测试实现和生产 fail-closed default，但真实 authority 与 production leak-scan 仍待外部证据。
3. 校验并收口已接入的 daemon-owned provider lifecycle、renew、restart、shutdown 和 generation recovery（含 bounded drain 与剩余 deadline 传递），复用已完成的 process table、group/Job cleanup 和 admission，并补齐 native evidence。
4. 在受控 identity host 上执行 daemon kill、provider crash-loop、PID reuse、vault redaction 和 orphan scan，生成可回读 evidence。

**解除标准**：故障后无 orphan process，run-as 不越权，真实 vault secret 零泄露，daemon restart 保持 budget/admission/ownership，三平台 production evidence 可复验；在真实 OS/KMS authority、EXT-01 identity host 和完整 Skill ownership/TOCTOU evidence 到位前不得标记完成。

### B-P0-03 MCP 生产传输未完成（W4）

**阻塞原因**：现有 stdio 和 legacy 明文 `http://` JSON-RPC 具备 allowlist、timeout/output limit 和 session cleanup；W4-L01 production transport 配置合同、W4-L02 HTTPS/TLS connector 仓库内实现、W4-L03 HTTP framing、W4-L04 Streamable HTTP session、W4-L05 增量 SSE 数据面、W4-L06 真实 abort/cleanup、W4-L07 supervisor-owned lifecycle 与 W4-L08 loopback 只读 server transport 均已接线，其中 W4-L01/W4-L02/W4-L07 仍在验证中。W4-L02 的仓库内代码缺口已关闭：单一绝对 deadline 覆盖 DNS、connect、TLS handshake、write、response head/body，逐次 I/O 重算剩余预算，真实 blocked-write/slow-drip 和 DNS deadline/capacity/address-limit 回归通过；Linux/macOS 逐级 `openat` dirfd 与 Windows `NtCreateFile` root-handle 相对遍历、同句柄类型/大小/读取及 final-path 校验关闭了受支持平台的祖先替换 TOCTOU。W4-L07 已让 daemon supervisor 在首次 I/O 前持有 registry 与 credential lease，graceful drain 关闭准入、回收 socket/session/reader、处理 cleanup 重试，并仅在 lifecycle credential owner、进程级 finalizer residual 及 v2 evidence 的 session/reader/cleanup-pending 三项残留全为 0 时完成；pending/failed credential lease 在生命周期最终析构时移交进程级有界 finalizer coordinator，以最多 4 个 worker 立即尝试新移交 owner，并仅在后续授权 drain deadline 内重试失败 owner，重启回归不复用 stale session。W3-L10 仍是 W4-L07 处于验证中的正式直接前置；registry 异常最终 Drop 只提供以同一起点、由最大 lifecycle timeout 限定总预算且每个 session 保留自身 timeout 上限的有界 owner 销毁兜底，进程退出或 SIGKILL 后死亡进程无法主动 DELETE，远端回收仍取决于命名 EXT-03 server 的 disconnect/TTL 合同。剩余缺口仅为 W4-L02 同提交三平台原生 evidence、W4-L07 的 EXT-03 SIGKILL/disconnect/TTL 外部验证，以及 W4-L09/L10 实跑 compatibility/external matrix；当前 compatibility 仍是 fixture。

**解决路径**：

1. 收口 W4-L02 production evidence：代码侧单调总 deadline、真实阻塞 write/slow-drip、Linux/macOS `openat` 与 Windows handle-based traversal/final-path 已完成；run `29788607021` 暴露的 writer inherited-FD 锁残留和 Windows terminal-frame 竞态，以及 run `29809011161` 暴露的 migration-lock duplicate-FD、macOS shutdown `EINVAL` 与 Windows CI fixture 预算问题，均已修复并具备严格回归。run `29811744322` 暴露的 create-new schedule 崩溃锁残留已改为稳定 anchor 上的 OS file lock，并由 run `29820120079` 的 Windows 原生 W1 evidence、Ubuntu/Windows core 与 W1 aggregate 成功验证；该 run 仅剩 macOS workspace 中依赖 150 毫秒墙钟上限的 reader join 测试假失败，现改为确定性 channel 屏障。待该测试修复提交的原生 Linux/macOS/Windows runner 与 W1 aggregate 全部成功，并回读 TLS、deadline、ancestor replacement/reparse、writer/migration unlock、failed-frame、abort、显式 timeout、schedule crash-takeover 与 reader join deadline evidence 后关闭该项。`file:` trust root 不得回退到当前工作目录或内联 PEM。
2. W4-L03/L04/L05/L06/L08 已完成 HTTP framing、Streamable HTTP session、增量 SSE/背压/断流检测、registry-owned 真实 reader/socket abort/DELETE/join 与 loopback 只读 server transport；保持 cleanup-pending 重试、server pre-handler gate 和 dangling=0 不变量。
3. W4-L07 的仓库内 owner/drain 已完成：保持 registry 与 credential authority 同生命周期；MCP cleanup 失败保留 session+lease，session 已移除后的 release 失败保留 retryable release task，最终析构把 pending/failed lease 移交进程级有界 finalizer coordinator，以 FIFO 队列和最多 4 个 worker 立即尝试新移交 owner，失败后的后续重试受授权 drain deadline 限制；保留包含 credential residual 门禁的 v2 零残留 evidence 与跨 generation 不复用不变量。W3-L10 仍须完成正式前置验证，并在 EXT-03 上验证进程退出/SIGKILL/disconnect/TTL 的远端回收合同。
4. 固定 EXT-03 server 名称、版本和 transport，实跑 stdio/HTTPS/auth/stream-abort matrix，并上传 measurement evidence。

**解除标准**：TLS/auth 错误稳定，stream abort 无资源泄漏，选定外部 server 实跑通过，静态 all-true fixture 不能满足 production gate。

### B-P0-04 备份恢复与升级交接缺少生产信任边界（W5）

**阻塞原因**：Alpha 已有本地 archive、plan-gated file mutation、rollback 和 operator confirmation，但 W5-L01 至 L12 均未完成。真实数据选择、持久 snapshot catalog、KMS 签名/AEAD、remote readback、可信 plan、target-root policy、crash-atomic restore 和 service-level handoff 尚未形成闭环。

2. 实现真实 source selector、atomic backup commit 和持久 snapshot catalog。
3. 使用 KMS signature 与 AEAD envelope，绑定 key version、artifact digest、issuer、expiry、target policy 和 nonce。
4. 实现受限 target root、crash-atomic restore、remote readback 和 W2-backed blue-green handoff。
5. 执行真实 dataset、KMS rotation/revoke、remote readback、candidate failure 和 pointer 不漂移 drill。

**解除标准**：未签名、过期、重放或越权 plan 在 mutation 前拒绝；失败 restore/upgrade 可恢复一致数据和 pointer，真实 artifact/service/remote evidence 可复验。

### B-P0-05 Production Closure 尚未实现（W10）

**阻塞原因**：W0 已完成 evidence 协议与 production policy 基础，但 DEC-01 尚未批准 production scope，W2-W9 的声明范围也没有全部真实 measurement。W10-L01 至 L11 尚未实现，当前 alpha `ready_with_external_blockers` 不能作为 publication decision。

```powershell
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
./scripts/validate-cli-json-contracts.ps1
./scripts/build-site-i18n.ps1
```
