# Eva-CLI 进程级高可用升级与恢复架构方案

## 1. 方案定位

本文定义 Eva-CLI 如何同时支持：

- 系统重启后的自动恢复。
- Runtime 自动升级。
- Supervisor 被杀死后的 Runtime 兜底拉起。
- Runtime 双活/蓝绿无缝切流。
- 进程内 EventBus 崩溃后的事件恢复。

必须先明确一个物理边界：**纯内存且从未持久化的事件，在进程崩溃后不可恢复**。因此本方案将“进程内 EventBus 可恢复”定义为：

```text
Recoverable In-Process EventBus
  = 进程内快速分发
  + 本地 Durable Event Log / WAL / Spool
  + ack / watermark / replay
```

也就是说，事件一旦被系统接受并向调用方确认，就必须先写入可恢复日志；真正从未落盘、也未被调用方确认的事件不承诺恢复。

核心结论：

- **OS Service Manager / 登录项 / launchd / systemd / Windows Service** 是系统重启自动恢复的根启动源。
- **Supervisor** 是 Runtime 主控制面，负责 Runtime 生命周期、双活切流、升级、回滚和恢复。
- **Runtime** 是业务执行面，内置 `SupervisorRecoveryGuard`，在 Supervisor 被杀死或失联时兜底拉起 Supervisor。
- **Runtime 双活** 采用 generation 模型：旧 Runtime 继续 drain 存量事件，新 Runtime 预热后接收新流量。
- **HardwareAdapter** 这类独占设备能力必须绑定 Runtime generation 和 per-device lease，避免双活期间两个 Runtime 同时写同一物理设备。
- **Lua Agent / Lua Capability 热更新** 是 Runtime 内部 generation swap；当权限边界、transport、状态 backend 或 sandbox 变化时，应升级为 Runtime generation 切流。
- **Ingress Gate** 负责按 generation、session、correlation_id 和切流策略分流。
- **Recoverable EventBus** 使用本地 Durable Event Log 让进程内分发具备崩溃恢复能力。
- **State Store** 保存 Supervisor、Runtime、generation、lease、watermark 和升级事务状态。
- 如果要求跨机器高可用，应把 Durable Event Log 替换为 Redis Streams、NATS、Kafka、RabbitMQ 或 PostgreSQL 队列。

## 2. 目标与非目标

### 2.1 目标

- 支持机器重启后自动恢复 Supervisor。
- 支持 Supervisor 根据 State Store 恢复 Runtime generation。
- 支持 Runtime 二进制自动升级。
- 支持新旧 Runtime 短时间双活，实现蓝绿/金丝雀/无缝切流。
- 支持旧 Runtime drain 存量任务，新 Runtime 接收新任务。
- 支持 Runtime 崩溃后由 Supervisor 自动拉起。
- 支持 Supervisor 被杀死或失联后由 Runtime 兜底拉起。
- 支持进程内 EventBus 接受事件后的崩溃恢复和重放。
- 支持幂等消费、dead letter、watermark、外部副作用 idempotency key。
- 支持审计、可观测和失败回滚。

### 2.2 非目标

- 不承诺恢复从未落盘、也未确认给调用方的纯内存事件。
- 不默认提供跨机器强一致事务。
- 不默认提供多个 Runtime 同时执行同一外部副作用的 exactly-once 语义。
- 不让 Lua Agent 触发进程升级、二进制替换、Supervisor 拉起或切流。
- 不让 Runtime 修改 Runtime release pointer。
- 不让 Runtime 成为完整控制面。
- 不把 OS service manager 设计成业务控制面。

## 3. 总体架构

```text
       OS Service Manager / Login Item / launchd / systemd / Windows Service
                                  |
                                  v
                         [Supervisor Process]
                                  |
              +-------------------+-------------------+
              |                                       |
              v                                       v
      [Runtime Gen N - active]             [Runtime Gen N+1 - candidate]
              |                                       |
              +-------------------+-------------------+
                                  |
                                  v
                        [Ingress Gate / Router]
                                  |
                                  v
                    [Recoverable In-Process EventBus]
                                  |
               +------------------+------------------+
               |                  |                  |
               v                  v                  v
      [Durable Event Log]   [State Store]      [Snapshot Store]
               |
               v
        WAL / spool / ack / watermark / replay
```

运行关系：

1. OS service manager 是系统重启恢复的根启动源。
2. Supervisor 启动后读取 State Store，恢复最近一次有效 Runtime generation。
3. Runtime generation 之间通过 generation id、lease、watermark 和 Ingress Gate 隔离。
4. Runtime 内部 EventBus 走内存快速路径，但事件被确认前必须写入 Durable Event Log。
5. 新 Runtime 升级时先作为 candidate 预热，通过 health check 后接收新流量。
6. 旧 Runtime 不再接收新流量，只处理已分配事件，完成后退出。
7. Supervisor 失联时，Runtime 内部 `SupervisorRecoveryGuard` 兜底拉起 Supervisor。

## 4. 核心组件

### 4.1 OS Service Manager

OS service manager 是机器重启后的根恢复机制。

可选形态：

- macOS `launchd`。
- Linux `systemd`。
- Windows Service。
- 用户登录项。
- 明确的外部启动器。

职责：

- 开机或登录后启动 Supervisor。
- Supervisor 异常退出后按策略重启 Supervisor。
- 不执行业务事件。
- 不管理 Agent、Adapter、MCP 或 EventBus。
- 不直接切 Runtime release。

### 4.2 Supervisor

Supervisor 是 Runtime 主控制面。

职责：

- 启动 Runtime generation。
- 监控 Runtime heartbeat 和 health check。
- 管理 Runtime lifecycle、lease 和 generation。
- 发起 Runtime 自动升级。
- 管理 Runtime release pointer。
- 管理 Runtime upgrade lock。
- 管理 Ingress Gate 切流状态。
- 触发 Runtime draining、snapshot、rollback。
- 机器重启后恢复上一次有效 Runtime generation。
- Runtime 崩溃后按 State Store 和 Event Log 拉起 Runtime。
- 自身被 Runtime 兜底拉起后 reattach 当前 Runtime。
- 记录审计日志。

Supervisor 不负责：

- EventBus Topic 匹配。
- Agent 业务调度。
- Adapter policy 判定。
- Lua State 执行。
- MCP tool 业务执行。

### 4.3 Runtime

Runtime 是业务执行进程。

职责：

- 运行 EventBus、Scheduler、AgentRuntime、AdapterRegistry 和 MCP。
- 暴露受控生命周期接口。
- 支持 `Running`、`Candidate`、`Draining`、`ControlDegraded` 等状态。
- 使用 Durable Event Log 接受、投递、ack 和重放事件。
- 向 Supervisor 汇报 heartbeat、health、版本、配置版本和 drain 进度。
- 运行 `SupervisorRecoveryGuard`。
- 支持 Supervisor reattach。
- 使用 control epoch 识别当前有效 Supervisor。

Runtime 不负责：

- 选择 Runtime release。
- 修改 Runtime release pointer。
- 发起 Runtime 升级。
- 回滚 Runtime release。
- 执行 Supervisor 自动升级。

### 4.4 SupervisorRecoveryGuard

`SupervisorRecoveryGuard` 是 Runtime 内部的最小兜底保活组件。

职责：

- 监听 Supervisor heartbeat。
- 判断 Supervisor 进程退出、control endpoint 断开或 heartbeat 超时。
- 将 Runtime 标记为 `ControlDegraded`。
- 按固定 Supervisor manifest 拉起 Supervisor candidate。
- 等待 Supervisor 重新 attach。
- 记录控制面丢失和恢复审计日志。

不负责：

- 修改 release pointer。
- 启动第二个 Runtime。
- 动态选择 Supervisor 版本。
- 执行 Runtime 升级或回滚。

### 4.5 Ingress Gate

Ingress Gate 是所有外部输入进入 Runtime 前的分流和状态门。

职责：

- 根据 Runtime generation 决定新事件进入哪个 Runtime。
- `Running` 时正常放行业务请求。
- `Candidate` 预热期间只允许健康检查、影子流量或金丝雀流量。
- `Draining` 时拒绝新业务请求或转发到新 generation。
- `ControlDegraded` 时按配置继续、收缩或暂停业务入口。
- 根据 session、correlation_id、topic、capability 和风险等级做 sticky routing。
- 防止升级期间新事件进入旧 generation。

默认切流策略：

| 事件类型 | 切流策略 |
| --- | --- |
| 新 session | 路由到新 generation |
| 旧 session 未完成链路 | 保持旧 generation sticky |
| 无状态分析任务 | 可切到新 generation |
| 高风险外部副作用 | 按 idempotency key 和旧 generation 状态决定 |
| 控制面请求 | 只交给 active Supervisor |

### 4.6 Recoverable In-Process EventBus

Recoverable EventBus 保留进程内低延迟分发，但补上可恢复日志。

```text
publish(event)
  -> assign event_id / sequence
  -> append Durable Event Log
  -> fsync or batch durability barrier
  -> publish to in-memory subscribers
  -> return accepted
```

职责：

- 为每个事件分配单调 sequence。
- 将事件写入 Durable Event Log。
- 将事件投递给 Scheduler、观测器和审计器。
- 保存 consumer ack / watermark。
- Runtime 崩溃后按 watermark 重放未确认事件。
- 支持 dead letter。
- 支持 compaction 和 retention。

关键语义：

- 已返回 `accepted` 的事件必须可恢复。
- 未返回 `accepted` 的事件可以由调用方重试。
- 单个事件可能被重放，Agent 必须幂等消费。
- 外部副作用必须使用 idempotency key。
- 如果选择 batch fsync，必须接受最后一个 batch 的风险窗口；高可靠模式应使用 strict fsync 或事务提交。

### 4.7 Durable Event Log

Durable Event Log 是本地持久化事件日志。

可选实现形态：

- append-only WAL 文件。
- SQLite 表。
- RocksDB / sled 类嵌入式 KV。
- 本地 spool 目录。
- 外部持久化队列。

最小数据结构：

```text
event_id
sequence
topic
target
payload
metadata
producer_id
generation_id
created_at
accepted_at
attempt
idempotency_key
checksum
```

Consumer 状态：

```text
consumer_id
generation_id
last_acked_sequence
inflight_event_ids
updated_at
```

### 4.8 State Store

State Store 保存高可用控制状态。

至少包含：

- supervisor instance id。
- supervisor pid / start time / endpoint。
- runtime generation id。
- runtime pid / start time / endpoint。
- active generation。
- candidate generation。
- draining generation。
- runtime release id。
- control epoch。
- upgrade transaction id。
- ingress routing state。
- event log high watermark。
- consumer watermarks。
- last known lifecycle state。

### 4.9 Snapshot Store

Snapshot Store 保存 Runtime 可恢复状态。

Snapshot 至少包含：

- snapshot id。
- runtime release id。
- generation id。
- config version。
- schema version。
- EventBus backend 类型。
- Durable Event Log position。
- Scheduler subscription table version。
- Agent status。
- Agent script version。
- Agent local state reference。
- processed event watermark。
- dead letter reference。
- Adapter health summary。
- MCP session recovery policy。
- audit log cursor。

Snapshot 不应包含：

- 明文密钥。
- Lua 未授权文件内容。
- 外部 provider session secret。
- 无法校验来源的临时内存指针。

## 5. 生命周期状态机

### 5.1 Supervisor 状态

```text
Starting
  -> RecoveringAfterBoot
  -> AcquiringLock
  -> AttachingRuntime
  -> Running
  -> PreparingRuntimeUpgrade
  -> StartingCandidate
  -> WarmingCandidate
  -> SwitchingIngress
  -> DrainingOldRuntime
  -> CommittingUpgrade
  -> Running
  -> RollingBackRuntime
  -> Failed
```

### 5.2 Runtime 状态

```text
Starting
  -> Recovering
  -> Candidate
  -> Running
  -> ControlDegraded
  -> Running
  -> Draining
  -> Snapshotting
  -> Stopping
  -> Stopped

Running
  -> Failed

Candidate
  -> Failed
```

关键状态：

| 状态 | 含义 |
| --- | --- |
| RecoveringAfterBoot | Supervisor 正在系统重启后恢复控制状态 |
| Candidate | 新 Runtime 已启动但尚未接收全部新流量 |
| WarmingCandidate | 新 Runtime 正在加载配置、Discovery、Agent、Adapter、MCP 和 EventBus 状态 |
| SwitchingIngress | Ingress Gate 正在把新流量切到新 generation |
| Draining | 旧 Runtime 不再接收新业务事件，正在处理存量工作 |
| ControlDegraded | Supervisor 失联，Runtime 业务面按策略继续或收缩 |

## 6. 系统重启自动恢复

系统重启后恢复链路：

```text
OS boot / user login
  -> OS Service Manager starts Supervisor
  -> Supervisor reads State Store
  -> Supervisor validates Release Store
  -> Supervisor opens Durable Event Log
  -> Supervisor identifies active / candidate / draining generation
  -> Supervisor starts required Runtime generation
  -> Runtime recovers Snapshot and EventBus watermarks
  -> Runtime replays unacked events
  -> Supervisor verifies health
  -> Ingress Gate resumes
```

恢复策略：

| 上次状态 | 恢复策略 |
| --- | --- |
| active generation clean | 启动 active generation |
| candidate warming | 丢弃 candidate，继续 old active 或重新开始升级 |
| ingress switching | 根据 commit marker 判断继续切流或回滚 |
| old draining | 恢复 old draining，继续 drain 或重放未确认事件 |
| upgrade committed | 启动 new active，清理 old generation |
| state uncertain | 保持入口关闭，进入人工恢复 |

关键要求：

- 升级事务必须有 prepare / commit / rollback marker。
- Ingress 切流状态必须持久化。
- Event Log 必须能定位 high watermark 和 consumer watermark。
- Runtime 启动后必须先恢复事件日志，再恢复业务入口。

## 7. Runtime 双活无缝切流

Runtime 双活采用 generation 模型，不是两个 Runtime 随机消费同一事件。

```text
Runtime Gen N
  -> active
  -> 处理旧 session 和已分配事件
  -> 进入 draining

Runtime Gen N+1
  -> candidate
  -> warmup / health_check / shadow traffic
  -> receive new traffic
  -> active
```

### 7.1 切流协议

```text
RuntimeUpgradeRequested
  -> Supervisor validates new release
  -> Supervisor starts Runtime Gen N+1 as candidate
  -> Candidate loads config / Discovery / Agent / Adapter / MCP
  -> Candidate opens Durable Event Log
  -> Candidate performs health_check
  -> Ingress Gate sends canary traffic
  -> Supervisor evaluates metrics
  -> Ingress Gate switches new traffic to Gen N+1
  -> Gen N enters Draining
  -> Gen N finishes assigned events or persists them
  -> Supervisor commits Gen N+1 active
  -> Supervisor stops Gen N
```

### 7.2 无缝语义

无缝切流要求：

- 新 Runtime 在接收真实流量前完成预热。
- Ingress Gate 切流前后都可用。
- 旧 Runtime drain 存量事件。
- 新事件进入新 Runtime。
- session sticky 路由保持一致。
- 共享 Durable Event Log 防止事件丢失。
- 外部副作用按 idempotency key 去重。

### 7.3 双活边界

双活期间允许：

- 新旧 Runtime 同时存活。
- 新 Runtime 处理新 session。
- 旧 Runtime 处理旧 session。
- 两个 Runtime 读取同一 Durable Event Log 的不同 consumer group / generation 分区。

双活期间禁止：

- 两个 Runtime 同时处理同一个非幂等事件。
- 两个 Runtime 同时持有同一个外部副作用 lease。
- 两个 Runtime 同时 claim 或写入同一个独占外接硬件设备。
- 旧 Runtime 接收已切到新 generation 的新事件。
- Candidate 未通过 health check 就成为 active。

## 8. Recoverable EventBus 恢复设计

### 8.1 发布语义

事件发布必须遵循：

```text
create Event
  -> validate schema / policy
  -> assign event_id and sequence
  -> append Durable Event Log
  -> durability barrier
  -> enqueue in-memory EventBus
  -> return accepted
```

如果 append 或 durability barrier 失败：

- 不返回 accepted。
- 调用方可重试。
- Runtime 不得把该事件视为已接受事件。

### 8.2 崩溃恢复

Runtime 崩溃后：

```text
Runtime restart
  -> open Durable Event Log
  -> read consumer watermarks
  -> load inflight_event_ids
  -> replay unacked events
  -> rebuild in-memory queues
  -> resume Scheduler
```

重放规则：

- `sequence <= last_acked_sequence` 默认不重放。
- `inflight_event_ids` 重放。
- dead letter 事件不自动重放，除非显式 retry。
- 外部副作用事件重放前检查 idempotency key。

### 8.3 Compaction

Durable Event Log 不能无限增长。

可清理条件：

- 所有 consumer watermark 均超过 sequence。
- 事件已超出 retention。
- 事件不在 dead letter retry 窗口。
- 对应 audit log 已落地。

### 8.4 不可恢复窗口

以下事件不可恢复：

- 写入 Durable Event Log 前进程崩溃，且调用方也未收到 accepted。
- 调用方本地创建但尚未发送到 Runtime 的事件。
- Runtime 未开启 durable barrier 时最后一个未刷盘 batch。

如需更强语义，必须启用 strict durability。

## 9. Supervisor 兜底恢复

Supervisor 兜底恢复由 Runtime 内部 `SupervisorRecoveryGuard` 触发。

```text
SupervisorHeartbeatLost
  -> Runtime enters ControlDegraded
  -> SupervisorRecoveryGuard validates supervisor launch descriptor
  -> SupervisorRecoveryGuard starts Supervisor candidate
  -> Supervisor candidate acquires singleton lock
  -> Supervisor candidate validates Runtime identity
  -> Supervisor candidate attaches Runtime control endpoint
  -> Supervisor candidate writes new control epoch
  -> Runtime accepts Supervisor control
  -> Runtime exits ControlDegraded
```

Supervisor candidate 必须获取 singleton lock 后才有效。Runtime 不直接认定自己拉起的进程就是 active Supervisor。

## 10. 回滚设计

### 10.1 Runtime 回滚

可自动回滚条件：

- previous Runtime release 仍存在。
- previous Runtime release manifest 校验通过。
- 当前 Snapshot 与 previous Runtime release 兼容。
- Durable Event Log schema 兼容。
- 状态 backend 没有执行不可逆 schema 变更。

不可自动回滚条件：

- 状态 schema 已不可逆迁移。
- Event Log schema 已不可逆迁移。
- 新版本已经执行不可逆外部副作用。
- Snapshot 与旧版本不兼容。
- Runtime release pointer 或升级锁状态不可信。

### 10.2 切流回滚

切流回滚根据 commit marker 判断：

| 阶段 | 回滚策略 |
| --- | --- |
| candidate warming | 停止 candidate，保留 old active |
| canary | 停止 candidate，新流量回 old active |
| switching ingress | 根据 routing marker 回滚或继续 |
| old draining | 新流量保持 new active，old 继续 drain |
| committed | 不自动回滚，走新升级事务 |

## 11. 配置设计

建议配置域：

```yaml
process:
  topology: supervisor_runtime_blue_green

service_manager:
  enabled: true
  restart_supervisor: true
  start_on_boot: true

supervisor:
  heartbeat_interval_ms: 1000
  heartbeat_timeout_ms: 5000
  singleton_lock: state/supervisor.lock

runtime:
  heartbeat_interval_ms: 1000
  heartbeat_timeout_ms: 5000
  singleton_lock: state/runtime.lock

  mode: blue_green

```

## 12. 安全边界

必须遵循：

- Lua 不得触发进程升级。
- Lua 不得触发 Supervisor 拉起。
- Agent 不得修改 release pointer。
- Runtime 不得修改 Runtime release pointer。
- Runtime 不得选择任意 Supervisor binary。
- Supervisor manifest 必须校验 checksum。
- Runtime 不得接受无效 control epoch 的 Supervisor 控制命令。
- 两个 Runtime 不得同时持有同一非幂等事件 lease。
- Event Log payload 必须按权限模型脱敏或加密敏感字段。
- Snapshot 不得保存明文密钥。
- 新 Runtime 启动前必须重新执行 policy 校验。

## 14. 风险与对策

| 风险 | 对策 |
| --- | --- |
| 系统重启后无人拉起 | OS service manager / 登录项启动 Supervisor |
| 纯内存事件崩溃丢失 | accepted 前写 Durable Event Log |
| Event Log 写入成功但内存投递失败 | 重启后按 watermark 重放 |
| 事件重复消费 | event_id 去重和幂等消费 |
| 外部副作用重复执行 | idempotency key 和副作用 lease |
| 双活重复处理同一事件 | generation 分区、consumer group、event lease |
| 切流中断 | Ingress routing marker 和 commit marker |
| Candidate 启动失败 | 停止 candidate，保留 old active |
| Old Runtime drain 超时 | timeout 后持久化、取消或死信 |
| Supervisor 被杀 | Runtime SupervisorRecoveryGuard 拉起 |
| Supervisor 与 Runtime 同时死亡 | OS service manager 恢复 Supervisor |
| Event Log 损坏 | checksum、segment recovery、dead letter |
| State Store 不一致 | audit log、transaction marker、人工恢复 |

## 15. 设计校验口径

方案成立需要满足：

- OS service manager 能在系统重启后启动 Supervisor。
- Supervisor 能从 State Store 恢复 Runtime generation。
- Runtime accepted 事件已写 Durable Event Log。
- Runtime 崩溃后能按 watermark 重放未 ack 事件。
- 新 Runtime 能作为 candidate 预热。
- Ingress Gate 能把新流量切到 candidate。
- 旧 Runtime 能进入 draining 并停止接收新事件。
- 双活期间同一非幂等事件不会被两个 Runtime 同时处理。
- Runtime 可以在 Supervisor 失联后拉起 Supervisor candidate。
- Supervisor candidate 必须通过 singleton lock 和 Runtime 身份校验。
- control epoch 能拒绝旧 Supervisor 控制命令。
- 回滚路径有明确 commit marker 和不可自动回滚条件。

## 16. 总结

Eva-CLI 的进程级高可用升级应落在 **OS service manager 根恢复 + Supervisor 主控 Runtime + Runtime SupervisorRecoveryGuard 兜底恢复 Supervisor + Runtime generation 双活切流 + Recoverable In-Process EventBus + Durable Event Log + Snapshot / State Store** 的组合上。

该设计支持系统重启后的自动恢复、Runtime 自动升级、Supervisor 被杀后的控制面恢复、双活无缝切流，以及进程内 EventBus 接受事件后的崩溃恢复。严格意义上，从未持久化的纯内存事件无法恢复；本方案通过“接受前先落 Durable Event Log”的方式满足可恢复语义。
