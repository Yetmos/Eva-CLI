# Eva-CLI 当前项目从零到完整实现实施计划

更新日期：2026-07-02

本文把当前仓库从“已有架构文档和 Rust workspace 骨架”推进到“核心 1.0 可发布”和“1.x 完整架构能力补齐”的实施步骤拆成可执行版本计划。它不是替代既有架构文档，而是把这些文档收束成工程实施顺序、模块功能清单、进度表和验收口径。

当前基线来自仓库实际状态：

- Rust workspace 已包含 19 个 crate：`eva-core`、`eva-config`、`eva-policy`、`eva-observability`、`eva-storage`、`eva-eventbus`、`eva-scheduler`、`eva-agent`、`eva-lua-host`、`eva-capability`、`eva-adapter`、`eva-mcp`、`eva-discovery`、`eva-memory`、`eva-hardware`、`eva-backup`、`eva-lifecycle`、`eva-runtime`、`eva-cli`。
- `eva-core` 已实现 Topic、ID、Event、Invoke、Capability 和结构化错误等基础契约。
- `eva-config` 已实现 `eva.yaml`、Agent/Adapter/Capability manifest 加载、跨文件校验和第一批测试。
- 其余多数 crate 已建立边界文件，但仍以 `RESPONSIBILITY` 占位为主，尚不能视为运行时行为完成。
- `cargo test --workspace` 已通过，当前测试集中在 `eva-core` 与 `eva-config`，共 66 个单元测试通过。

## 1. 版本路线总览

![Eva-CLI 从零到完整实现版本路线](../assets/full-implementation-roadmap.zh-CN.svg)

- **核心 1.0**：让 CLI、配置、EventBus、Scheduler、AgentRuntime、Lua host、受控 capability 和最小示例形成稳定闭环。1.0 不追求所有高阶能力完整，而是保证核心承诺可运行、可测试、可诊断。
- **完整 1.x**：在 1.0 之上补齐 Adapter/MCP/Skill、Discovery、Memory/Knowledge、Hardware、Backup/Snapshot、Lifecycle/Supervisor、发布和运维能力。

## 2. 模块实施顺序

![Eva-CLI 模块实施顺序](../assets/full-implementation-module-sequence.zh-CN.svg)

推荐实施顺序遵循四条规则：

1. 先实现稳定契约，再实现副作用。
2. 先跑通一条窄闭环，再扩展 Adapter、MCP、硬件和恢复。
3. `eva-runtime` 只做组合根，下层 crate 不反向依赖 runtime。
4. 每个版本只扩大一类行为面，并补齐测试和文档。

| 模块 | 功能 | 先后顺序 | 当前进度 | V0.2 交付 |
| --- | --- | --- | --- | --- |
| `eva-core` | Topic、Event、Invoke、ID、Capability、Error | 1 | 完成 | 维持零依赖，补齐文档与 API 注释 |
| `eva-config` | `eva.yaml`、manifest 反序列化 | 2 | 完成 | 字段命名与 JSON Schema 对齐 |

## 6. V0.4 最小 Runtime 闭环计划

![Eva-CLI 最小 Runtime 闭环](../assets/full-implementation-runtime-loop.zh-CN.svg)

目标：用最小实现证明架构主链路成立。

| 模块 | 功能 | 先后顺序 | 当前进度 | V0.4 交付 |
| --- | --- | --- | --- | --- |
| `eva-storage` | in-memory event log / state store | 1 | 骨架 | 支持测试中的 accepted/ack/watermark |
| `eva-eventbus` | publish/subscribe/dead letter | 2 | 骨架 | in-memory EventBus 单元测试 |
| `eva-scheduler` | subscription 和 mailbox 投递 | 3 | 骨架 | target、exact、wildcard 路由测试 |
| `eva-agent` | queue、lifecycle、event handler | 4 | 骨架 | Agent 能消费一个事件 |
| `eva-lua-host` | 最小 Lua on_event | 5 | 骨架 | 沙箱 Lua 执行、`ctx.emit`、`ctx.tools` 占位 |
| `eva-capability` | builtin capability | 6 | 骨架 | 一个无副作用 capability 返回结构化结果 |
| `eva-runtime` | 服务组合 | 7 | 骨架 | runtime builder 装配上述服务 |
| `eva-cli` | `eva run --example basic` | 8 | 骨架 | 端到端运行示例 |
| `examples/basic` | 最小项目 | 9 | 未开始 | 事件进入、Agent 处理、capability 返回 |

闭环验收：

1. CLI 构造 `/input/user` typed event。
2. EventBus 发布事件并保留最小恢复元数据。
3. Scheduler 按 TopicPattern 找到目标 Agent。
4. AgentRuntime 从 bounded inbox 消费事件。
5. Lua `on_event(event, ctx)` 运行。
6. Lua 调用一个受控 Rust builtin capability。
7. capability 返回结构化结果。
8. Runtime 输出 trace、audit、成功或结构化错误。
9. 集成测试覆盖成功路径和至少一个失败路径。

## 14. 每个版本的固定实施流程

每个版本按同一节奏推进：

1. 明确版本目标、非目标和可交付用户行为。
2. 补或更新契约测试，先锁定新增行为边界。
3. 实现最小代码切片，优先复用现有 crate 边界。
4. 增加单元测试，跨模块行为增加集成测试。
5. 更新中文详案、英文摘要和 i18n manifest。
6. 运行 `cargo test --workspace`。
7. 对 CLI 行为补充手动验证命令和示例输出。
8. 更新版本进度表和剩余风险。

## 15. 风险与控制表

| 风险 | 影响版本 | 控制方式 |
| --- | --- | --- |
| 继续增加抽象但没有端到端闭环 | V0.2-V0.4 | V0.4 前禁止扩展高阶能力，先跑通 `examples/basic/` |
