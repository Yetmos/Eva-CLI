Eva-CLI

多Agent调度中枢

面向可热更新多 Agent 协作的开源 CLI 运行时方案。用 Rust 守住权限、调度和可靠性边界，用 Lua 承载可快速迭代的业务 Agent，并通过 Topic EventBus 连接 MCP、Skill、外部工具、本地模型和硬件能力。

Eva 开源核心项目，与盈利产品 EvaLauncher 同源但保持清晰区分。

Why Eva-CLI

为可控的 Agent 协作而设计

Rust Core

系统边界由 Rust 托管

调度、权限、沙箱、密钥、超时、审计、恢复和外部能力接入都由 Rust 负责，避免业务脚本绕过安全边界。

Lua Agents

业务逻辑支持热更新

Agent 意图识别、局部状态转换、工具编排和结果映射由 Lua 承载，适合快速调整工作流而不重新编译主程序。

Topic EventBus

用事件解耦多 Agent

Agent 通过路径式 Topic 协作，不共享隐式全局状态，让路由、扇出、异步任务和观测链路更清晰。

Adapter Layer

统一接入外部能力

Claude、Codex、Gemini、本地模型、内部服务、MCP server、workflow skill 和硬件设备都通过受控 Adapter 暴露能力。

Use Cases

适合落地的使用场景

01

本地 AI Agent 自动化中枢

把用户输入、项目事件、CLI 任务和后台工作流统一投递到 EventBus，由多个 Agent 分工完成分析、计划、审查、修复建议和文档生成。

02

内部工具与 Agent 网关

将内部 HTTP 服务、自研 CLI、本地模型和团队工作流封装为 Adapter，通过 manifest、schema、policy 和审计控制可调用能力。

03

MCP 双向集成平台

内部 Agent 可以调用外部 MCP server；Eva-CLI 也可以作为 MCP server，把受控 Agent、Adapter 和 Lua capability 暴露给外部客户端。

04

项目知识库与长期记忆

将项目文档、设计方案、运行手册、Agent 私有记忆和系统级长期记忆纳入 ContextBuilder，为多 Agent 协作提供可溯源上下文。

05

硬件和边缘设备自动化

通过 HardwareAdapter 接入 USB、串口、BLE、网络设备和厂商 SDK 设备，让 Agent 订阅硬件事件并调用受控业务能力。

Architecture

一条清晰的运行链路

Eva-CLI 将发现、调度、执行和扩展拆成独立层次：发现层负责扫描和校验，调度层负责 Topic 路由，执行层运行 Lua Agent，扩展层通过 Adapter 接入外部能力。Lua 只表达业务意图，Rust 负责权限、协议、生命周期和审计。

  1. 发现能力扫描 Agent、Adapter、MCP、Skill 和硬件 manifest。
  2. 发布事件CLI、API、UI 或外部系统写入 Topic Event。
  3. 调度 AgentScheduler 按 Topic、target、优先级和负载投递到私有队列。
  4. 调用工具Lua Agent 通过 Rust Tool Layer 调用受控 Adapter。
  5. 观测恢复审计、指标、状态快照、重放和 generation 切流保障可恢复性。

Compare

与 LangGraph 的定位差异

LangGraph 和 Eva-CLI 解决的问题有重叠，但抽象层级不同。LangGraph 更适合把 LLM 应用建模为显式状态图、节点和边；Eva-CLI 更关注本地环境里的多 Agent 运行时、受控能力接入、热更新和进程级可靠性边界。

LangGraph

更像可持久化的 Agent 工作流图

官方文档将 LangGraph 定位为 orchestration runtime，重点能力包括 durable execution、streaming、human-in-the-loop 和 persistence。它通过 State、Nodes、Edges 组合复杂工作流，让图在状态演进中运行。

Eva-CLI

更像受控的多 Agent 能力运行时

Eva-CLI 以 Rust Runtime、Topic EventBus、Scheduler、Lua Agent 和 AdapterRegistry 为核心，把 MCP、Skill、CLI、HTTP、本地模型、内部服务和硬件设备统一纳入受控能力层。

维度 LangGraph Eva-CLI
核心抽象

StateGraph、节点、边、条件路由和 checkpoint，适合描述一个 LLM workflow 如何一步步演进。

Topic EventBus、Agent 私有队列、Scheduler、Adapter capability，适合描述多个 Agent 和外部能力如何协作。

状态模型

状态是图运行的核心数据结构；checkpointer 保存线程内图状态，store 可承载跨线程长期记忆。

Agent 局部状态、系统记忆库、知识库和事件日志分层管理；EventBus 只负责通知和链路，不作为业务状态存储。

扩展方式

通常在图节点中接入模型、工具、函数、LangChain 组件或自定义代码。

通过 Adapter manifest、schema、policy 和 transport 接入 Claude、Codex、MCP server、workflow skill、HTTP 服务和硬件设备。

运行边界

重点在工作流编排、持久化执行、人类介入、调试和部署能力。

重点在本地 CLI runtime、沙箱、权限、密钥隔离、审计、热插拔、进程恢复和 Runtime generation 切流。

热更新

更常见的调整方式是修改图定义、节点逻辑或部署版本。

目标是让 Lua Agent、Lua Skill、Tool handler 和 MCP handler 支持 generation swap，失败时保留旧版本。

更适合

确定性或半确定性的 LLM 工作流、带 checkpoint 的长任务、人类确认节点、多轮 agent loop。

本地多 Agent 中枢、内部工具网关、MCP 双向集成、项目知识库、长期记忆、硬件与边缘设备自动化。

选择 LangGraph：当你要把一个应用流程清晰建模成图，并且需要成熟的 checkpoint、interrupt、streaming 和 LLM workflow 生态。

选择 Eva-CLI：当你需要一个本地多 Agent runtime，把不同 Agent、工具、MCP、Skill 和硬件能力放进统一权限与审计边界。

组合使用：LangGraph 可以作为某个 Adapter 或外部 Agent 被 Eva-CLI 调用；Eva-CLI 负责运行时边界，LangGraph 负责具体工作流图。

LangGraph 描述参考其官方文档： Overview、 Graph API、 Persistence。

Principles

不是无限制插件系统

Lua 不直接执行 shell、不读取密钥、不连接任意外部服务。

MCP、Skill、CLI、HTTP 和硬件调用都必须经过 AdapterRegistry、Router、schema 和 policy。

EventBus 负责协作链路，不承担隐式全局状态或长期记忆存储。

热更新覆盖业务脚本和可热加载 manifest 字段，权限扩大和 transport 变化需要更严格的 runtime 切换。
