Open source agent runtime

Eva-CLI

A controlled runtime for multi-Agent automation

Eva-CLI uses Rust for reliable runtime boundaries and Lua for hot-reloadable Agent behavior, then connects external Agents, MCP servers, skills, memory, knowledge bases, and hardware through a unified Topic EventBus.

Rust runtime boundary Lua hot reload Topic EventBus
Read the docs Architecture overview Development progress Discussion Feedback
Execution model Controlled automation path
  1. Rust Core Owns policy, memory, recovery, and process boundaries.
  2. Lua Agents Hot-reloads behavior while keeping capability access explicit.
  3. Adapters Routes CLI, MCP, HTTP, skills, and hardware through permissions.

V1.6.1-alpha adds the durable backend schema and migration-lock baseline while keeping release checks executable.

Development progress

From architecture scaffold to the V1.6 durable runtime baseline

Open roadmap
Current stage

V1.6.1-alpha durable backend baseline is implemented

The repository now has the V1.5 release-hardening and GHCR delivery surface plus the first V1.6 durable backend contract: schema-versioned backend.manifest, stable events/state/tasks/audit/artifacts layout, read-only verification, and migration locking for future durable EventBus, task, audit, and artifact stores.

  1. Done
    Architecture, module partitioning, and configuration

    System boundaries, runtime model, adapters, memory, discovery, hardware, recovery, website, docs, configuration, and module partitioning are in place.

  2. Done
    Runnable V1.0 runtime and task diagnostics

    The in-memory basic loop, CLI quickstart, local task status/log/cancel diagnostics, and core CI checks are implemented.

  3. Done
    V1.1 to V1.5.1 controlled capability, release gates, and GHCR delivery

    External capability diagnostics, memory context, hardware plans, backup/lifecycle checks, release hardening commands, the source user manual, and GitHub Packages container delivery are implemented.

  4. Now
    V1.6.1-alpha durable backend baseline

    Schema-versioned durable backend layout, read-only verification, migration locking, and in-memory compatibility backend are implemented for the next durable runtime slices.

  5. Later
    Signed installers, provenance, and real apply paths

    Future releases add provider process execution, durable stores, destructive apply gates, signed installers, OS packages, and provenance bundles.

Next implementation work
  • Wire durable EventBus publish/ack/fail/dead-letter redrive onto the new backend layout.
  • Promote task, audit, and artifact stores to durable adapters and add restart recovery evidence.
  • Keep signed installers, OS packages, provenance bundles, and destructive apply gates behind later V1.10/V1.11 milestones.
